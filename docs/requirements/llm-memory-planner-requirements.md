## 2. 产品目标与边界

LLM Memory Planner 是一个纯前端的大模型推理显存规划工具。用户配置一份单机、多机或 Prefill/Decode（PD）分离部署，工具按节点、rank 和运行阶段计算显存峰值，并与单卡实际可用显存比较。

工具必须回答：

1. 给定机器和并行策略，模型权重能否在每张物理设备上完成放置并加载。
2. 权重加载后，在扣除运行时必要显存预算的情况下，剩余显存可以支持多大的 KV/状态缓存和上下文。
3. 当前配置中每个 rank 的权重、缓存及其他显存分项是多少，最坏物理设备和瓶颈是什么。
4. 当前配置属于“可运行”“OOM”或“可能运行”中的哪一种，并在可能运行时标出是否临界。
5. 如何调整上下文、运行请求数、卡数、节点数或并行拓扑以满足显存约束。
6. 满足显存约束的最小卡数、节点数和并行拓扑是什么。

本项目只评估显存，不估算吞吐、TTFT、TPOT、网络时间、排队时间或其他性能指标。Prefix cache 命中率可以作为场景信息或机会容量展示，但不得提高有保证的显存容量。

### 2.1 首要用户问题

首版必须优先、分两级回答以下问题：

#### 问题一：权重能否加载

根据模型 checkpoint、量化格式、设备、机器拓扑、TP/PP/DP/EP/CP 和 rank 放置，计算每张物理设备上的权重及加载所需的已知固定项，定位最坏设备，并回答该并行策略能否放置模型权重。

结果至少包含：

- 各 rank 持有的层、专家和 tensor shard；
- 各物理设备的权重显存；
- 最大权重设备及剩余显存；
- 权重无法放置时的最小卡数或拓扑调整建议；
- checkpoint resident bytes 与加载/repack 临时峰值的证据边界。

“模型权重能否加载”必须拆成两个独立结论：

- **权重可放置**：只判断模型加载完成后的常驻权重及明确的权重派生 tensor，能否按当前并行策略和 rank 放置进入每张物理设备。该结论不包含 runtime、KV pool、Graph 和稳态 activation。
- **服务可启动**：在权重可放置的基础上，进一步考虑加载临时副本、量化解包或 repack、runtime/通信初始化、必要内存池、Graph capture 及其他启动期共存项。

权重不可放置时，服务必然不能启动；权重可放置不代表服务一定能够启动。两个结论必须同时展示，不能合并为一个模糊的“能加载”。

只有模型、固定 checkpoint、tensor metadata、权重放置规则和缓存机制均已内置并完成适配时，才允许计算“权重可放置”。未内置模型阻止规划，不提供结构估算或启发式降级。服务启动所需经验项缺少适用上界时，服务启动结论为“可能，需要实测”。

#### 问题二：权重加载后支持多大上下文

在每张设备的权重常驻显存基础上，扣除 runtime、Graph、activation、通信、allocator、安全余量及其他必要预算，把可用空间换算为 KV/latent/indexer/状态缓存容量，并回答上下文上限。

上下文容量必须同时给出以下口径，不能只给一个含义不明的 token 数：

- 最大总缓存 tokens 或 blocks；
- 单请求、并发为 1 时的最大上下文；
- 用户指定最大运行请求数时，每请求可保证的最大上下文；
- 用户指定 context 时可支持的最大运行请求数；
- 因 page/block 取整、后端限制和模型最大位置长度造成的约束。

这里的“最大 token”是缓存容量约束下的 context capacity，可以表达为单并发最大长度，也可以表达为多并发请求占用的总 token 容量。工具必须展示 context 与并发的容量关系，而不能默认“最大上下文”只指单请求：

```text
sum(每个运行请求实际缓存 tokens 向 block/page 取整后的容量)
<= 当前部署可用的总缓存容量
```

在所有请求采用相同上限的保守规划口径下，可以近似展示：

```text
保证总缓存 tokens >= 并发请求数 * 每请求最大 context（含 block/page 取整）
```

模型最大位置长度仍限制单请求 context，但不等于多并发总缓存 token 上限。

当经验预算只有上下界时，分别输出“保证容量”和“可能容量”。保证容量使用经验项保守上界计算；可能容量可以使用经验项下界或典型值展示，但必须标明需要实测。

#### 问题三：OOM 能否通过增加机器解决

当当前配置 OOM 时，工具必须判断增加机器或设备是否能够通过改变并行拓扑降低最坏设备显存，并给出最小可行节点数、卡数和并行策略候选。

增加机器本身不会自动降低原设备显存。只有新增设备被纳入同一个模型副本，并通过 TP/PP/EP/CP 或其他受支持分片规则重新放置权重、缓存或状态，才可能解除单副本 OOM。增加 DP 副本通常只增加服务副本和总容量，不降低单副本的每卡权重或单请求最大上下文，因此不得作为解决单副本 OOM 的建议。

工具必须按 OOM 原因解释扩容是否有效：

- 权重 OOM：评估更大的 TP、PP、EP 或其组合是否能降低最坏设备权重；
- KV/状态缓存 OOM：评估后端实际支持的 KV/cache 分片是否随 TP、PP、CP 等扩展，不能默认按总卡数等分；
- Graph、activation、runtime 或通信 OOM：只有存在适用校准或明确实现规则时，才判断扩卡后的变化；否则输出“可能，需要实测”；
- 设备内不可分片或固定复制项已经超限：增加机器无法解决，必须更换单卡显存更大的设备、改变 dtype/量化或后端配置；
- PD 部署：分别判断 Prefill 池和 Decode 池，只扩容发生瓶颈的池。

多机方案只评估显存可行性。跨机通信是否满足性能要求不在本项目结论范围内，但后端不支持的跨机拓扑必须由兼容性检查阻止。

## 3. 用户与运行环境

- 面向需要规划和复现大模型推理部署的工程人员。
- 界面为中文专业桌面工作台，保留标准英文术语和后端参数名。
- 支持最新版 Chrome、Edge，最低视口宽度为 1280 px。
- 首版不支持移动端。
- 应用是纯前端静态应用，不请求 `/api/*`。
- 所有服务配置、命令、manifest、trace 和日志只在浏览器本地处理，不上传、不执行。

## 4. 统一部署场景

产品使用一份统一部署配置描述以下场景，不将其割裂为独立工具：

- 单机部署；
- 同构多机部署；
- PD 分离部署。

一个普通部署池由一台或多台机器组成，每台机器包含 8 张或更多同类型加速卡。需要支持多机，并为其他每机卡数保留自定义能力。

同一个池内的机器规格一致，不会出现一个池内混用 NVIDIA 卡和 P800，也不会出现同一个 Prefill 池或 Decode 池包含不同类型机器。

## 5. 输入要求

### 5.1 模型

- 首版只支持内置、已适配并通过验收的固定模型和 checkpoint，不支持用户自定义模型、用户自定义 checkpoint、上传 `config.json` 或上传 Tensor Manifest。
- 任一内置模型必须固定 repository、revision、checkpoint 格式、量化格式、tensor metadata 生成版本、元数据日期及证据来源。
- 内置 checkpoint metadata 必须能确定每个 tensor 的名称、shape、物理 dtype、数据字节数、所属 shard，以及共享/重复关系；不能只保存 checkpoint 总字节或 index 中的 `total_size`。
- 每个内置模型必须具有已验证的模型结构、tensor placement、cache strategy，以及适用的框架、平台和版本兼容性证据。任一必要模块未适配时，该模型或组合必须标记为 `unsupported`，不得启发式估算。
- 新模型通过产品版本更新加入。适配流程必须补充固定 checkpoint metadata、模型结构规则、权重放置规则、缓存公式、框架/平台兼容性和 golden/实测验证；通用容量计算和 UI 不因单个模型增加专用条件分支。
- 原始 trace/log、标准 trace JSON 和手工显存项只允许用于已支持模型及兼容组合的校准或实测覆盖，不能用于绕过模型适配要求。

### 5.2 负载

负载必须包含：

- 最大 input tokens；
- 最大 output tokens；
- 最大 context；
- 最大运行请求数（max running requests）；
- 最大 batched tokens。

不得把未知的必填负载字段静默当成零或默认成一个看似可用的值。使用默认值时必须明确展示默认值及来源。

### 5.3 硬件

硬件配置必须包含：

- 设备型号；
- 机器数；
- 每机卡数；
- 单卡物理显存；
- 单卡实际可用显存。

首版内置设备预设包括：

- NVIDIA H20 96GB；
- H20-3e；
- NVIDIA A100 80GB；
- NVIDIA A800 80GB；
- 昆仑芯 P800；
- 自定义设备。

设备预设只表达显存规划所需的规格和已校准能力，不承担性能建模。

### 5.4 并行拓扑与 rank 放置

校验模式必须输入具体的：

- TP；
- PP；
- DP；
- EP；
- CP；
- rank 到节点、设备及并行角色的放置关系。

搜索模式必须输入：

- 各并行维度允许范围；
- 可用机器和每机设备约束；
- rank 放置约束；
- 允许搜索的节点数和卡数范围。

不能只用总显存或平均显存判断可运行性。必须计算到 rank，并考虑不均匀分片、页或 block 向上取整、复制状态以及最坏 rank。

### 5.5 PD 分离

- Prefill 和 Decode 是同一部署配置中的两个池。
- 每个池内机器和设备规格一致。
- P/D 两池可以使用不同平台，例如 `P=NVIDIA, D=P800` 或 `P=P800, D=NVIDIA`。
- 分别配置和计算 P/D 的模型、负载、机器、设备、并行拓扑、rank 放置及运行阶段。
- P/D 显存独立判定，不把两池设备显存合并。
- PD connector 必须计入发送、接收、staging、格式转换及其他同时驻留的显存。
- Connector 支持已验证预设、自定义 P/D 显存项以及 trace 实测覆盖。
- 不估算 connector 或网络性能。

### 5.6 校准输入

以下输入均可用于经验项校准：

- 原始 vLLM、SGLang 及平台分支 trace/log；
- 工具定义的标准 trace JSON；
- 用户手工输入的分阶段、分 rank 显存项。

实测 trace 的适用值优先于校准模型，校准模型优先于通用预算或未知状态。原始日志默认只保留在本地；导出报告默认只包含脱敏后的证据摘要，用户明确选择后才可附带原始日志。

## 6. 服务配置导入

### 6.1 支持格式

- vLLM、SGLang 及平台分支的直接 CLI 命令；
- `docker run` 命令；
- Docker Compose YAML；
- Kubernetes YAML，包括 workload、容器 command/args、环境变量、资源、节点选择和副本配置。

### 6.2 映射目标

导入器将以下内容映射到统一部署配置：

- 模型、revision、checkpoint 和精度；
- 负载；
- 显存预算参数和 Graph 配置；
- TP/PP/DP/EP/CP；
- rank 放置；
- PD 配置；
- 镜像、硬件资源、节点和副本。

如果服务命令或 manifest 引用的模型、revision 或 checkpoint 不在内置支持目录中，导入器仍应保留并展示已解析参数，但必须将模型标记为 `unsupported`，阻止显存计算和可执行配置导出。

### 6.3 解析结果与安全

导入结果必须独立展示：

- 已识别参数；
- 未识别参数；
- 冲突参数；
- 缺失必填项；
- 使用的默认值及来源。

关键输入未确定时阻止计算，不静默猜测。未识别但无冲突的参数必须无损保留。

- 只进行静态解析，绝不执行命令、shell 展开、容器或 manifest。
- `$VAR` 允许用户手工补值，并保留原变量表达式。
- `$(...)`、反引号、重定向、管道及其他可执行 shell 结构只保留并标记为未解析，绝不执行。
- 自动识别并脱敏 token、密码、Secret、registry 凭据及其他敏感值。

## 7. 服务配置导出

支持导出：

- 直接 CLI；
- `docker run`；
- Docker Compose；
- Kubernetes。

导出内容是可运行的最小显存复现配置，而不是生产部署全套配置。内容必须包括：

- 镜像；
- 模型挂载占位；
- command、args 和 env；
- GPU/XPU 资源；
- 节点数和副本；
- 并行拓扑和必要 rank 放置信息；
- PD 场景所需的最小组件。

不生成 Service、Ingress、Secret、认证、监控或性能调优配置。

后端、平台或拓扑兼容性检查未通过时，不生成声称可执行的命令。可以输出带明确阻断原因的草稿，但不得标为可运行。

导入后重新导出要求语义等价；允许格式、引号和参数顺序变化，但所有显存相关参数必须等价。未知参数必须保留并标注，不能在转换中丢失。

## 8. 显存总账与计算语义

### 8.1 运行阶段

至少分别计算：

- 初始化与 Graph capture；
- Prefill；
- Decode。

只有同一阶段实际并存的显存项才相加。每个 rank 取各阶段总量中的最高值作为峰值，部署结论取所有节点和 rank 的最坏值。

### 8.2 显存项

显存总账至少包含：

- 权重；
- KV cache；
- latent cache、indexer cache 和模型状态缓存；
- 激活；
- CUDA/XPU Graph；
- runtime、通信库和 allocator；
- 安全余量；
- connector 显存；
- 用户或 trace 定义的自定义项。

每个显存项必须携带来源、证据等级、适用范围、公式或校准版本以及是否与其他项同阶段共存，不能只输出一个不可解释的总数。

### 8.3 理论项与经验项

- 权重和 KV/状态缓存应在输入完整时按确定性公式计算。
- KV 计算必须覆盖 MHA/GQA/MQA、MLA、稀疏或压缩布局、滑动窗口和混合层、分页/block 取整及后端实际分片规则。
- 权重计算必须考虑 checkpoint 实际 dtype、量化元数据、未量化参数、共享参数、PP/TP/EP 分片和复制项。
- 激活、Graph、runtime、allocator 和通信等经验项必须使用版本化校准、用户预算或实测值。
- Graph 不假定存在跨模型、后端和设备通用的可靠闭式公式。
- 校准范围外不得无提示外推；应标为未知或证据不足。

## 9. 判定规则

默认安全余量为单卡实际可用显存的 5%，允许用户调整。

输入校验、后端兼容性和显存结论必须独立展示：

1. 输入校验决定是否具备计算条件。
2. 兼容性结论判断模型、后端、平台、版本和拓扑是否有支持证据。
3. 显存结论只判断各物理设备的保守显存边界是否满足物理显存和安全余量；rank 用于确定显存归属和定位最坏设备。

显存判定不以经验项的单点估计值为依据，而使用保守上下界：

```text
设备显存下界 = 确定性显存 + 经验项保守下界
设备显存上界 = 确定性显存 + 经验项保守上界
```

显存状态定义如下：

- **可运行**：所有物理设备的显存上界均满足安全余量。所有非微小显存项必须有确定性公式、适用校准上界、用户保守预算或同配置实测覆盖。
- **OOM**：任一物理设备的显存下界已经超过物理显存。下界必须来自确定性公式、适用的保守校准下界或同配置实测，不允许使用“典型值”判定 OOM。
- **可能运行**：理论确定项尚未证明 OOM，但上下界跨越容量边界，或存在无法建立可信边界的经验项；结论必须提示需要在真实机器上验证。
- **临界**：作为“可能运行”的风险标签使用，表示没有超过物理显存的确定证据，但无法满足安全余量；不再作为与“可能运行”并列的确定性结论。

`可运行`和 `OOM` 是只针对显存的结论，前提是输入有效且后端兼容；它们不保证模型功能、性能、网络或服务稳定性。“可能运行”不是兼容性结论，也不得被描述为已验证可运行。

导入同配置实测后，可以额外标记“实测可运行”或“实测 OOM”。实测标签和规划器判定必须分开展示，避免把校准推断伪装成当前部署已经运行过。

## 10. 输出要求

### 10.1 主结果

- 每个节点、rank、阶段的显存总账；
- 最坏节点、rank 和阶段；
- 瓶颈显存项；
- 距离物理显存和安全阈值的剩余或超出量；
- 输入校验、兼容性、显存结论三类独立状态；
- 证据完整度和未知项。

### 10.2 规划结果

- 当前约束下最大 context；
- 当前约束下最大运行请求数；
- 满足约束的最小卡数、节点数及拓扑；
- 对 context、运行请求数、卡数、节点数和并行拓扑的可解释调整建议。

### 10.3 分享与报告

- 非敏感紧凑配置的分享 URL；
- 机器可读 JSON；
- 完整可复现 Markdown 报告。

Markdown 报告至少包含：

- 全部有效输入和默认值；
- 模型、checkpoint、后端及适配器版本；
- 硬件、拓扑和 rank 放置；
- 分阶段、分 rank 显存总账；
- 三类状态与最终显存结论；
- 生成的服务配置；
- 校准边界、适用范围及边界验证证据；
- 证据摘要；
- 未知项、自定义项和安全余量。

## 11. 校准、正确性与验收

### 11.1 正确性目标

- 权重每 rank 理论值与实测值偏差不超过 2%。
- KV/状态缓存偏差不超过 1%，或不超过一个实际 page/block，取较宽松者。
- 激活、Graph、runtime 等无法可靠理论计算的项目，在 H20/P800 上建立版本化保守边界或由实测覆盖。
- 取消经验分项 10% 和总峰值 5% 的点估计误差目标。校准的目标是补足理论无法确定的显存项，并支持可靠分类，而不是追求完全准确的峰值预测。
- 在正式验收集合内，规划器给出的“可运行”必须能在真实机器上以声明的配置和负载完成显存验收，不发生设备 OOM。
- 在正式验收集合内，规划器给出的“OOM”必须能在真实机器上以声明的配置和负载复现设备 OOM。
- 无法满足上述单向可靠性要求的配置必须输出“可能运行”，交由真实机器验证，不能为了提高结论覆盖率强行归类。

指标只在对应证据前提满足时验收：

- 权重 2% 指标要求使用内置固定 checkpoint 的完整 tensor metadata、已验证并行切分规则和自动生成的 rank 放置。
- KV/状态缓存指标要求具备已验证的后端 layout/cache spec、量化附加数据、rank 切分和 page/block 取整规则；未知布局不得按标准 KV 公式代替。
- checkpoint 文件总字节、理论 rank resident tensor bytes 和 runtime 实测 resident bytes 是三种不同 measurement kind，报告必须分别标注；实测值可以覆盖理论值，但不能混为同一个原始指标。

校准不要求穷举所有调度时序。校准域只需要记录会实质改变显存边界的条件，包括：

- 运行请求数或 batch；
- batched tokens；
- context；
- Graph capture 模式和尺寸；
- TP/PP/DP/EP/CP；
- 权重、激活和缓存 dtype；
- prefix cache；
- chunked prefill。

每条校准必须记录样本范围、保守下界、保守上界、边界生成方法、后端/平台/版本和验证数据。范围外不得外推为“可运行”或“OOM”。典型值可以用于展示大概显存，但不能用于确定性判定。

### 11.2 正式验收模型

- MiniMax-M3，最高优先级；
- Llama 3.1 8B；
- DeepSeek-V3.2；
- Qwen3.5 9B；
- Qwen3.5 35B-A3B；
- Gemma 4 E2B；
- Gemma 4 31B。

MiniMax-M3 和 DeepSeek-V3.2 必须覆盖原始精度及至少一种有实测证据的量化预设。大型模型允许使用可信量化和多机部署完成验收。

每个正式验收模型在 H20、P800 上至少有一条后端路径实际运行并完成校准。其他后端组合只能以有证据的 `unsupported` 完成兼容性验收；环境阻塞或校准失败不算发布验收通过。

PD 验收必须覆盖：

- `P=NVIDIA, D=P800`；
- `P=P800, D=NVIDIA`。

PD 只验证显存，不验证性能。

### 11.3 命令往返验收

往返测试必须覆盖：

- CLI、Docker、Compose 和 Kubernetes；
- vLLM、SGLang 及至少一个平台分支；
- 变量和手工补值；
- 未知参数保留；
- 参数冲突；
- 缺失必填项；
- 敏感信息识别和脱敏；
- 显存相关参数的语义等价。

## 13. 非目标

首版明确不包含：

- 吞吐、TTFT、TPOT、排队、网络时延或成本性能预测；
- 自动执行用户命令、shell 展开、镜像或 Kubernetes manifest；
- 自动拉取私有模型、镜像或 Secret；
- 生产级 Service、Ingress、认证、Secret、监控和性能调优配置；
- 同一个部署池内混用 NVIDIA 和 P800；
- 移动端适配；
- 服务端存储、账号系统或 `/api/*`。
