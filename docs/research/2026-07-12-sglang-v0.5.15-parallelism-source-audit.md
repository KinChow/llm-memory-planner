> 状态：方案设计输入，非最终产品设计
>
> 审计日期：2026-07-12
>
> 审计对象：SGLang `v0.5.15`
>
> 固定 commit：`f63458b5beaceabbd9d749b9fc956370e1b649e6`
>
> 目的：确认 SGLang 并行参数在进程、模型组件、KV cache、调度和运行时显存中的真实语义，为后续显存规划器设计提供依据

## 1. 执行摘要

SGLang 的并行配置不能归纳成“模型权重依次除以 TP、PP、DP、EP”。源码显示，它至少包含三层不同语义：

1. **部署副本层**：普通 DP 会启动 `DP` 份独立的 `TP x PP` distributed world；每份有独立 scheduler 和 KV cache。
2. **组件并行层**：开启 DP Attention 后，不再启动 `DP` 份 world，而是在同一个 `TP x PP` world 中，将相同 TP 轴分别分解为 Attention 和 MoE 两套坐标。
3. **运行阶段层**：Prefill CP、DCP、chunked prefill、CUDA Graph 和 PD disaggregation 会改变 token、KV、临时 buffer 或 Graph 的放置，但不一定增加 worker。

核心关系如下：

```text
普通 DP：
  总 GPU worker = DP x TP x PP
  每个 DP 副本的 distributed world = TP x PP

DP Attention：
  总 GPU worker = TP x PP
  Attention: TP = Attention-DP x Attention-CP x Attention-TP
  MoE:       TP = MoE-DP x EP x MoE-TP

任一 distributed world 内：
  world_size = TP x PP
  global_rank = TP x pp_rank + tp_rank
```

由此可得：

- `DP rank`、`TP rank`、`PP rank` 不是应当分别计算后相加的显存实体；显存最终落在同一个物理 GPU worker 上。
- `DP Attention` 不是普通 DP，也不是简单地让 attention 权重除以 `DP`。它将请求和 KV 分到独立的 Attention-DP 请求/缓存容量域，同时让 FFN/MoE 处理跨 DP gather 后的 token；每个容量域内部仍由 Attention-TP、Attention-CP 和 PP workers 协同执行。
- `EP` 和 `MoE-DP` 不增加独立 GPU 乘数；它们是在现有 TP world 内重新解释 routed expert 的权重和 token 归属。
- 标准 KV cache 的 head 分片使用 `Attention-TP`；attention 权重是否同样使用它，取决于模型构造是否向 QKV layer 传入有效 Attention-TP。GQA/MQA 在有效 TP 超过 KV head 数时会复制 KV head。
- PP 按 layer stage 放置权重与 KV；各 stage 的可用 cache 容量最终取最小值。
- Graph、activation、DP gather/scatter、MoE A2A、DCP 及 PD staging 都是不能从 checkpoint 权重和 KV payload 公式直接推导的运行时项。

因此，显存规划器必须先由固定版本 adapter 解析实际 worker、group、scheduler 和组件放置，再计算每张物理 GPU 的账本。本审计只给出上述设计事实，不在此确定最终产品 schema。

## 3. 参数不是最终运行事实

### 3.1 主要公开字段

SGLang v0.5.15 暴露的并行相关输入包括：

| 参数 | 默认值 | 初始含义 |
|---|---:|---|
| `--tp-size` / `--tp` | 1 | distributed world 内的 tensor-parallel 宽度 |
| `--pp-size` / `--pp` | 1 | pipeline stage 数 |
| `--dp-size` / `--dp` | 1 | 普通 DP 副本数，或 DP Attention 下的 Attention-DP 数 |
| `--enable-dp-attention` | false | attention 使用 DP/CP/TP 分解，FFN/MoE 使用另一套分解 |
| `--attention-context-parallel-size` | 1 | Attention-CP 宽度 |
| `--decode-context-parallel-size` | 1 | DCP 宽度 |
| `--enable-prefill-cp` | false | 启用 prefill context parallel |
| `--cp-strategy` | null | `zigzag` 或 `interleave` |
| `--expert-parallel-size` / `--ep` | 1 | routed expert 的 EP 宽度 |
| `--moe-data-parallel-size` | 1 | MoE-DP 宽度 |
| `--moe-dense-tp-size` | null | MoE 模型中 dense MLP 的 TP 宽度 |

来源：`python/sglang/srt/server_args.py:814-917,1762-1772,1849-1855`。

### 3.2 resolved 参数会改变用户表面配置

`enable_dp_attention`、`attn_cp_size`、`ep_size`、`moe_dense_tp_size` 等字段带有 `resolvable=True`，模型家族、硬件、backend 和 A2A post-process 会参与默认解析。后续代码大量使用 `self._resolved()` 或 `resolved_view(self)`，而不是原始字段。

这意味着导入命令时必须同时保存：

- 用户明确输入的值；
- exact-version adapter 解析后的有效值；
- 自动改写的原因和证据；
- 因缺少硬件或模型信息而暂时无法解析的字段。

不能只把命令行参数原样送入通用公式。源码证据见 `python/sglang/srt/server_args.py:900-915,1765-1771,1849-1854` 以及 `python/sglang/srt/arg_groups/overrides.py` 的 resolution passes。

### 3.3 关键拓扑约束

exact tag 在 post-process 中至少执行以下约束：

```text
TP % (DP x Attention-CP) == 0       # DP Attention/Attention CP
TP % MoE-DP == 0
EP x MoE-DP <= TP
EP > 1 时通常要求 EP x MoE-DP == TP
Attention-CP != MoE-DP 时，只允许 MoE-DP == 1
DP Attention 时 TP % DP == 0
moe_dense_tp_size in {None, 1, TP}
MoE-DP > 1 时 PP == 1
普通 DP > 1 且未启用 DP Attention 时，仅支持单节点
```

此外，部分 backend 有更严格限制。例如 FlashInfer MoE A2A 要求 DP Attention 开启且 `DP == TP`。来源：`python/sglang/srt/server_args.py:4848-4888,4899-4904,5142-5143,6577-6588`。

这些只是全局约束；模型、attention backend、量化和 speculative decoding 还会增加组合限制。

## 4. Worker、rank 和 process group

### 4.1 普通 DP

普通 DP 会循环 `dp_rank in range(dp_size)`，为每个 DP rank 启动一套 tensor-parallel group。每套 group 内再按 PP 和 TP 启动 scheduler process。因此：

```text
总 GPU worker = DP x TP x PP
每个 DP 副本 = 独立的 TP x PP distributed world
```

Ray 路径也显式返回相同公式。来源：

- `python/sglang/srt/managers/data_parallel_controller.py:276-334`
- `python/sglang/srt/ray/engine.py:105-112`

不同 DP 副本不在同一个模型 world 内做 dense weight sharding。每个副本加载自己的模型权重，创建自己的 scheduler 和 KV pool，controller 在副本之间选择请求。

### 4.2 DP Attention

DP Attention 路径只调用一次 `launch_tensor_parallel_group`，在这一个 TP/PP world 内为不同 TP rank 推导 Attention-DP rank，并为每个 DP rank准备独立请求端口：

```text
总 GPU worker = TP x PP
```

这不是 `DP x TP x PP`。源码明确说明“Data parallelism reuses the tensor parallelism group”。来源：`python/sglang/srt/managers/data_parallel_controller.py:456-485,524-540`、`python/sglang/srt/ray/engine.py:105-112`。

### 4.3 world rank

无论普通 DP 的某一个副本，还是 DP Attention 的折叠 world，每个 target-model worker 都按下式初始化 distributed environment：

```text
world_size = TP x PP
global_rank = TP x pp_rank + tp_rank
```

普通 DP 的多个副本分别拥有各自的 world；DP 不进入该 world_size 公式。来源：`python/sglang/srt/model_executor/model_runner.py:1301-1323`。

### 4.4 物理 rank 放置

worker 启动先推导当前 node 覆盖的 PP rank，再推导该 PP stage 覆盖的 TP rank：

```text
pp_size_per_node = max(PP / nnodes, 1)
nnodes_per_pp_rank = max(nnodes / PP, 1)
tp_size_per_node = TP / nnodes_per_pp_rank
```

GPU id 随 `pp_rank`、`tp_rank` 和 `gpu_id_step` 计算。默认含义是相邻 TP rank 尽量在同一节点；当一个 TP group 跨节点时，连续 TP rank 依次分布。来源：`python/sglang/srt/managers/data_parallel_controller.py:504-516,543-548`。

规划器不能只检查“总卡数够不够”，还必须按这一自动放置规则检查 `nnodes`、每节点 GPU 数、TP/PP 整除关系及每张物理卡承载的 stage/component。

### 4.5 实际建立的 groups

一个 `TP x PP` world 中可能建立：

- TP group；
- DCP group；
- Attention-CP group；
- Attention-TP group；
- MoE-DP group；
- MoE-EP group；
- MoE-TP group；
- PP group；
- PD multiplexing 开启时的重复 prefill TP group。

它们大量复用同一组物理 rank，并不是各自占用一套 GPU。来源：`python/sglang/srt/distributed/parallel_state.py:2073-2313`。

## 5. 同一 TP 轴的两套组件分解

### 5.1 Attention 分解

源码计算：

```text
Attention-TP = TP / Attention-DP / Attention-CP
TP = Attention-DP x Attention-CP x Attention-TP
```

其中：

- 未启用 DP Attention 时，`Attention-DP = 1`；
- 启用后，`Attention-DP = dp_size`；
- `Attention-CP = attn_cp_size`；
- Attention-TP group 由每个 DP/CP slice 内的连续 rank 组成；
- Attention-CP group 由相同 Attention-TP 坐标、不同 CP 坐标的 rank 组成。

来源：`python/sglang/srt/distributed/parallel_state.py:2137-2207`。

这套坐标决定 attention 模块应使用的权重分片、head 数、KV payload 和 context split，不能直接用全局 TP 代替。

### 5.2 MoE 分解

同一个 TP 轴又独立计算：

```text
MoE-TP = TP / EP / MoE-DP
TP = MoE-DP x EP x MoE-TP
```

MoE-DP、MoE-EP 和 MoE-TP group 分别建立。当 `Attention-CP > MoE-DP` 时，MoE-DP group 可能直接复用 Attention-CP group，以便进入 MoE 前共享 token。来源：`python/sglang/srt/distributed/parallel_state.py:2209-2293`。

所以同一张 GPU 同时拥有两套坐标，例如：

```text
物理 worker (pp=0, tp=5)
  Attention 坐标: dp=1, cp=0, attn_tp=1
  MoE 坐标:       moe_dp=0, ep=2, moe_tp=1
```

两套坐标服务不同模型组件，不能被压成单一的 `parallel divisor`。

### 5.3 PP group

PP group 使用相同 TP 位置、跨 PP stage 的 rank 组成，rank 间距等于每个 stage 的 TP world 宽度。来源：`python/sglang/srt/distributed/parallel_state.py:2295-2313`。

因此一个 worker 的完整身份至少包含 `pp_rank`、全局 `tp_rank`，以及从 TP rank 派生的 Attention 和 MoE 两套坐标。

## 6. 权重放置语义

### 6.1 标准 TP linear

SGLang 的基础实现与 Megatron 风格一致：

- `ColumnParallelLinear` 沿 weight 输出维切分；每 rank 持有 `output_size / tp_size`。
- `RowParallelLinear` 沿 weight 输入维切分；bias 不切分。
- 是否 gather/reduce 输出由具体 layer 参数决定。

来源：`python/sglang/srt/layers/linear.py:291-382,1339-1424`。

这说明即使在 dense Transformer 中，也不能把全部 tensor bytes 统一除以 TP：norm、bias、部分 scale、router 和其他小 tensor 可能复制；量化 loader 还会引入 packing、padding 和 scale tensor。

### 6.2 QKV 与 GQA/MQA 复制

`QKVParallelLinear` 按传入的 TP rank/size 切 Q/K/V。只有模型构造明确传入有效 Attention-TP 时，下面的权重结论才按 Attention-TP 成立；否则基础层默认使用全局 TP。若该模型路径中的有效 TP 不小于 `total_num_kv_heads`，每个 rank 至少保留一个 KV head，并计算对应的 KV head replicas。来源：`python/sglang/srt/layers/linear.py:893-969`。

Qwen3-MoE 的模型实现同样使用有效 `attn_tp_size` 并设置：

```text
local KV heads = max(1, total KV heads / Attention-TP)
```

来源：`python/sglang/srt/models/qwen3_moe.py:441-489`。

显存影响：

- 已适配模型的 Q projection 通常随 Attention-TP 下降；
- 已适配模型的 K/V projection 和标准 KV cache 仅在 KV head 可继续分片时下降；
- 当有效 Attention-TP 超过 KV head 数后，继续增加它不再降低每卡 K/V 权重和 KV bytes/token；
- DP Attention 减小有效 Attention-TP，可能改变 KV 复制倍数。

模型 adapter 必须验证 QKV layer 构造实际使用的 rank/size，不能只因启用了 DP Attention 就假定所有模型权重自动改用 Attention-TP。

### 6.3 embedding 和 lm_head

`VocabParallelEmbedding` 按 vocabulary 维度切分，并为可整除和 LoRA 预留做 padding；`ParallelLMHead` 复用该机制。开启 DP Attention 后，一些模型可选择在 Attention-TP group 内做 vocabulary parallel，避免跨 DP group gather，但该行为由模型和 `enable_dp_lm_head` 能力决定。来源：

- `python/sglang/srt/layers/vocab_parallel_embedding.py:185-290,547-625`
- `python/sglang/srt/server_args.py:911-916`

因此 embedding/head 的实际分片宽度也必须读取模型构造参数，不能假设永远等于全局 TP。

### 6.4 PP 按层而不是平均切权重

模型通过 `start_layer`、`end_layer` 和 PP rank 只构造当前 stage 的 transformer layers；首尾 stage 还会分别持有 embedding、final norm、lm_head 或其他非对称模块。每个 stage 的层数与额外模块不同，不能使用 `total checkpoint bytes / PP` 代替 tensor 级放置。

planner 必须根据固定模型 adapter 的 layer mapping 和 checkpoint tensor metadata 计算每个 PP stage，尤其要保留：

- 首 stage embedding；
- 末 stage final norm/lm_head；
- 非均匀 layer 数；
- MTP/speculative/draft 额外模块；
- tied/untied embeddings；
- 模型专有 indexer、shared expert 或 projection。

PP world 与 layer-local KV 的 runtime 证据见 `python/sglang/srt/model_executor/model_runner.py:1301-1323` 以及 cache 预算对 `start_layer/end_layer` 的使用：`python/sglang/srt/model_executor/pool_configurator.py:125-137,222-233`。

### 6.5 routed experts、shared experts 和 dense MLP

SGLang 对 routed experts 使用 `MoE-DP x EP x MoE-TP`：

- EP 分配不同 routed experts；
- MoE-TP 切一个 expert 内部的矩阵；
- MoE-DP 复制相同 expert shard，但各自处理不同 token；
- router/gate、shared experts、shared-expert gate、norm 和 scale 仍需按模型代码独立判断。

`moe_dense_tp_size` 只允许 `None`、`1` 或全局 `TP`。当它为 `1` 时，dense MLP 不做 tensor sharding，每个参与的 local replica 持有完整矩阵；具体复制域仍由 DP Attention 坐标和模型构造决定。来源：`python/sglang/srt/server_args.py:1849-1855,6584-6588`、`python/sglang/srt/layers/dp_attention.py:262-276`。

A2A backend 还会影响 resolved EP 和 workspace。例如 FlashInfer A2A 要求 `DP == TP` 且开启 DP Attention；DeepEP 的通信 buffer 会进入 Graph/runtime 预算。来源：`python/sglang/srt/server_args.py:1773-1785,3675-3684,5142-5143`。

## 7. 普通 DP 的调度和显存语义

普通 DP controller 为每个 `dp_rank` 启动一套 `TP x PP` scheduler group，并在不同 worker endpoint 间进行 round-robin、total-requests、total-tokens 或显式 rank 路由。来源：`python/sglang/srt/managers/data_parallel_controller.py:276-334,605-652`。

由此得到：

- 每个 DP 副本加载完整的一份按 TP/PP 切分后的模型；
- 每个 DP 副本拥有独立 KV cache 容量和请求集合；
- 单个请求只路由到一个 DP 副本；
- 服务总 max tokens 可以聚合副本容量，但单副本/单请求容量不能把所有副本 KV pool 相加；
- 最坏显存仍应在所有副本的物理 worker 中逐卡判断。

普通 DP 的主要显存变化是“复制整个副本”，而不是降低单卡权重或单请求 KV。机器数增加只有在部署形式确实增加普通 DP 副本、扩大 TP/PP world 或改变硬件时才改变结果。

## 8. DP Attention 的完整执行语义

### 8.1 请求和 KV ownership

DP Attention 为每个 Attention-DP 请求/缓存容量域建立独立 endpoint，controller 将请求路由到其中一个域。一个域包含具有相同 `attn_dp_rank` 的 Attention-TP、Attention-CP 和 PP workers；这些 worker 各自持有本地权重与 KV shard，共同完成该域的请求。单个请求不会跨 Attention-DP 域占用 KV。

来源：

- `python/sglang/srt/managers/data_parallel_controller.py:456-485,605-652`
- `python/sglang/srt/layers/dp_attention.py:245-259`

所以 DP Attention 下的容量至少应分开报告：

- 每个 Attention-DP 请求/缓存容量域的 max tokens；
- 整个部署组的 aggregate max tokens；
- 一个请求能够使用的单域最大 context。

### 8.2 Attention 局部计算

Attention 模块在本地 Attention-DP 请求上执行，权重按 Attention-TP 切分，Prefill CP 或 DCP 再作用于 context/token 维。Attention-DP 之间不是共享同一请求的普通 tensor parallel。

这正是 `Attention-TP = TP / DP / Attention-CP` 的用途。DP Attention 的显存变化包含：

- attention weight 分片宽度改变；
- 每个 Attention-DP 容量域及域内 worker 的 KV ownership 改变；
- 每 rank 处理的本地 attention token 改变；
- GQA/MQA 的 KV head 复制倍数可能改变。

### 8.3 进入 FFN/MoE 前后的 gather/scatter

不同 DP rank 的 token 需要进入共享的 FFN/MoE component topology。源码提供：

- all-gather/all-gatherv 或 all-reduce，将本地 token 填入 global token buffer；
- MoE/FFN 对 global buffer 执行计算；
- reduce-scatter/reduce-scatterv 后返回各 Attention-DP rank；
- 某些配置还需要 Attention-TP all-gather。

来源：`python/sglang/srt/layers/dp_attention.py:472-691`。

因此 DP Attention 会产生与 token 数、hidden size、padding mode、dtype 和 collective backend 相关的持久或临时通信 buffer。它不能只通过权重和 KV 公式完成显存计算。

### 8.4 chunked prefill 自动改写

开启 DP Attention 后，SGLang 断言 `TP % DP == 0`，并将 resolved `chunked_prefill_size` 除以 `DP`。来源：`python/sglang/srt/server_args.py:4899-4905`。

这个改写直接改变 prefill 单迭代 token 上限和 activation 峰值。显存规划器应展示用户值和 effective 值，不能静默沿用导入命令中的原始 chunk size。

### 8.5 CUDA Graph 和 runtime reserve

DP Attention 不是纯通信模式。默认 runtime reserve 会额外加入：

```text
decode_graph_max_bs x DP x 3
若 decode_graph_max_bs > 300，再加 max_bs x DP x 1.5
```

代码注释说明该额外项来自操作 padding 以及大 Graph batch 下 allocator/实现的高显存。来源：`python/sglang/srt/server_args.py:3647-3658`。

因此 DP Attention 的 Graph/runtime 预算必须按 SGLang 版本、模型、backend 和 capture shapes 校准，不能从理论 KV 公式推断。

### 8.6 模型支持不是全局布尔值

CLI help 仍写“DeepSeek-V2 和 Qwen 2/3 MoE”，但 exact tag 中更多模型实现已引用 `is_dp_attention_enabled`、Attention-TP group 或 DP communicator。反过来，代码引用并不证明所有 attention backend、量化、speculative、PP、PD 和设备组合都可用。

因此 planner 对 DP Attention 应使用证据等级，而不是仅按模型架构名返回 true/false。

## 9. EP、MoE-DP 和 A2A 的显存影响

### 9.1 EP

EP 在现有 TP world 内分配 routed experts，不增加 `x EP` 的 GPU 数。每个 rank 持有其 EP slice 的专家，同时还可能持有共享 experts、router 和 dense component。更大的 EP 通常降低 routed-expert weight bytes/rank，但不会等比例降低所有 MoE 层权重。

### 9.2 MoE-DP

MoE-DP 让不同 group 复制 expert shard、处理不同 token。它可能提高每卡 expert 权重相对 EP-only 的复制量，但减少本地 dispatch token；实际关系由 `EP x MoE-DP x MoE-TP = TP` 决定。

### 9.3 MoE-TP

MoE-TP 切单 expert 的矩阵。若 `EP x MoE-DP == TP`，则 MoE-TP 为 1；若 EP 较小，则一个 expert 仍可在多个 rank 上 tensor-shard。

### 9.4 通信 workspace

不同 A2A backend 会分配不同 dispatch/combine buffer。SGLang 默认显存比例计算甚至为 capture 进 CUDA Graph 的 DeepEP A2A buffer额外预留 2 GiB。来源：`python/sglang/srt/server_args.py:3675-3684`。

显存规划器必须将 routed-expert weights 与 A2A/runtime workspace 分开：前者可以根据 checkpoint tensor 和 component placement 确定，后者需要版本化预算或实测证据。

## 10. KV cache 放置和容量

### 10.1 标准 MHA/GQA/MQA

标准模型的 per-token cache cell 使用：

```text
local KV heads x (K head dim + V head dim)
x 当前 PP stage 的 KV layer 数
x KV dtype bytes
```

其中 local KV heads 按有效 `Attention-TP` 计算，而不是外部 TP。来源：`python/sglang/srt/model_executor/pool_configurator.py:175-182,251-268`。

这会保留 GQA/MQA 的复制饱和：`Attention-TP > total KV heads` 后 per-rank KV heads 不再下降。

### 10.2 MLA、DSA、MiniMax sparse 和 hybrid cache

SGLang 的 pool configurator 对不同 cache family 有不同计算路径：

- MLA：latent rank + decoupled RoPE；
- DSA：在 MLA 外增加 indexer KV/scale；
- MiniMax sparse：主 K/V pool 加 sparse indexer，且部分层可 key-only；
- sliding/full hybrid：分别计算 full 与 SWA pool；
- Mamba/linear attention：还存在 state cache 和按请求 slot 的固定项；
- FP4 KV：payload 外增加 scale buffer；
- EAGLE/standalone/DFLASH：target pool 外计入 draft cache。

来源：`python/sglang/srt/model_executor/pool_configurator.py:118-268,284-518`。

因此 planner 的模型 cache strategy 必须与 exact SGLang model/backend adapter 对齐，不能只有通用 `2 x layers x heads x head_dim`。

### 10.3 cache pool 从剩余显存反推

SGLang 启动时先加载权重并测量剩余显存，再扣除 `mem_fraction_static` 对应的 runtime slack，得到可分配给 KV pool 的 bytes。若权重已经挤占静态预算，会直接拒绝启动。来源：`python/sglang/srt/model_executor/model_runner_kv_cache_mixin.py:103-139`。

cache token capacity 为：

```text
floor(available cache bytes / bytes per token / page_size) x page_size
```

来源：`python/sglang/srt/model_executor/pool_configurator.py:270-281`。

这说明服务能否启动不只由“权重小于物理显存”决定，还要求 runtime reserve 后至少能建立有效 KV pool 和其他必需对象。

### 10.4 PP 的最小容量

各 PP stage 的层数和剩余显存可不同。SGLang 对各 stage 的 token capacity 做 `MIN` all-reduce，整个副本使用最小值。来源：`python/sglang/srt/model_executor/model_runner_kv_cache_mixin.py:1196-1223`。

规划器应同时显示：

- 各 PP worker 的理论/预算 cache capacity；
- 限制整个副本的最坏 stage/rank；
- 对齐 page 后的有效 max tokens。

### 10.5 普通 DP 与 DP Attention 的容量聚合

- 普通 DP：每个完整副本有独立 KV capacity；总服务容量是副本容量之和，但单请求上限取一个副本。
- DP Attention：每个 Attention-DP 请求/缓存容量域有独立 capacity，域内由多个 worker 的本地 KV shard共同实现；部署 aggregate 只可对各域的有效 capacity 求和，单域和单请求限制必须单独保留。

二者在总数上都可能表现为“多份 cache”，但 worker 数、attention weight 放置和 FFN/MoE 通信完全不同。

## 11. Prefill CP 和 DCP

### 11.1 Prefill Attention CP

`--enable-prefill-cp` 必须指定 `--cp-strategy`，可选 `zigzag` 或 `interleave`。它沿 prefill sequence/token 维在 Attention-CP ranks 间分工，并复用现有 TP world，不额外乘 worker 数。来源：`python/sglang/srt/server_args.py:884-898,4817-4855`。

它会改变：

- 每 rank prefill attention/activation token；
- Attention-TP 宽度，因为 `TP = DP x CP x Attention-TP`；
- CP communication 和 LSE/intermediate workspace；
- 模型与 backend 的有效组合；
- 部分模型的 CUDA Graph 策略。

例如 DSA/MLA prefill CP 会禁用 prefill CUDA Graph；PD decode-only deployment group 不允许开启该 prefill CP。来源：`python/sglang/srt/server_args.py:3895-3907,3920-3937`。

### 11.2 DCP

DCP 在每个 TP group 内建立大小为 `dcp_size` 的连续 subgroup，要求：

```text
DCP >= 1
TP % DCP == 0
v0.5.15 仅 CUDA/HIP 平台允许 DCP > 1
```

来源：`python/sglang/srt/distributed/parallel_state.py:2056-2071,2114-2135`。

DCP 不增加 worker，也不改变 checkpoint 权重分片；它复用 TP ranks 切 decode context/KV 处理。不过具体 cache layout、backend 和 speculative 支持需要按模型组合判断，不能只根据 `TP % DCP == 0` 宣称可用。

### 11.3 CP 不能在产品中折成一个无语义数字

SGLang v0.5.15 至少存在：

- Prefill Attention CP；
- Decode Context Parallel；
- DP Attention 中的 Attention-DP；
- DSA/MLA 模型专有 CP 自动配置。

它们复用物理 rank 的方式、作用阶段、KV ownership 和 Graph 兼容性不同。后续设计必须保留语义，不能只存 `CP=2`。

## 12. Activation、CUDA Graph 和 runtime

### 12.1 chunked prefill 和 activation

SGLang 根据 GPU memory、TP 和模型/backend 自动选择 chunked prefill size；H20/H200 分支默认 `8192`，但 DP Attention 又会将它除以 DP。来源：`python/sglang/srt/server_args.py:3500-3556,4899-4905`。

Prefill activation 峰值至少受以下因素影响：

- effective chunked prefill tokens；
- batch 内 decode token 是否共存；
- Attention-TP/CP/DP 的 token 分布；
- PP stage 的 layer/module；
- attention、MoE 和量化 kernel；
- overlap/two-batch-overlap；
- collective buffer 是否与计算重叠。

因此 activation 不能使用跨模型、跨 backend 的单一 hidden-size 常数保证准确。

### 12.2 Graph capture shapes

decode Graph 默认根据 GPU memory 和 TP 生成 batch-size 列表；prefill Graph 的 max size通常由 chunked prefill size决定，MLA 默认另有限制。Graph 按较大 shape 优先 capture 以复用 memory pool。来源：

- `python/sglang/srt/server_args.py:3558-3621`
- `python/sglang/srt/model_executor/runner/decode_cuda_graph_runner.py:690-752`

Graph 显存与 capture size 列表、模型执行路径、DP padding、A2A backend、allocator 和 PyTorch/CUDA 版本相关。它应由版本化校准或实测覆盖，而不是仅按最大 batch 做线性推导。

### 12.3 默认 static reserve 是 heuristic

当用户未设置 `mem_fraction_static` 时，SGLang 会用一系列 MiB 级 heuristic 预留 metadata、activation slack、decode Graph、prefill Graph、TP/PP 和 A2A buffer；检测到显存大于 60 GiB 时，最低 reserve 为 10 GiB。随后再换算成 `mem_fraction_static`。来源：`python/sglang/srt/server_args.py:3623-3689`。

这些数值是框架启动预算，不是可以保证真实峰值的理论公式。规划器应把它们作为 exact-version runtime evidence，并用 H20 实测校准上下界。

## 13. PD disaggregation

### 13.1 两个部署组各自完整启动

SGLang 使用：

```text
--disaggregation-mode prefill
--disaggregation-mode decode
```

分别启动 prefill-only 和 decode-only server，connector backend 默认 Mooncake，也可由 exact tag支持的 backend 选择。来源：`python/sglang/srt/server_args.py:2270-2289`。

Prefill、Decode 侧各自加载模型、建立 worker topology、cache pool、runtime 和 scheduler。二者显存不能相加成一张“PD 每卡显存”，必须分别判断各部署组。

### 13.2 Decode 侧不是只有 KV payload

Decode 请求经历 pre-allocation、transfer、waiting 和 running 四个生命周期。Decode 侧会预先为接收请求分配 KV，并额外维护 request-to-token table；preallocated/transfer 请求可以超出 `max_running_requests` 对应的运行槽。来源：`python/sglang/srt/disaggregation/decode.py:1-19,107-144,275-320`。

所以 Decode 显存至少包含：

- 本地常驻模型和 decode runtime；
- 目标 KV/cache pool；
- preallocated/transfer/running 请求的 metadata 和映射表；
- connector 注册与通信资源；
- 可选 staging；
- speculative/Mamba/HiCache 等组合的额外状态。

### 13.3 Prefill 侧发送现有 cache buffer

Prefill manager 将目标 KV pool（以及存在时 draft KV pool）的 pointer、长度、item size 和辅助 metadata buffer 注册给 transfer backend。来源：`python/sglang/srt/disaggregation/prefill.py:152-228`。

传输量本身不是额外 GPU 常驻显存；是否存在额外发送 buffer 要看 connector 和 staging 路径，不能一律把完整 KV 再复制一份。

### 13.4 异构 TP staging

当 Prefill 与 Decode 的有效 Attention-TP 不同，可通过 `SGLANG_DISAGG_STAGING_BUFFER=1` 启用 GPU staging：

- 将分散 KV head slice gather 到连续 GPU buffer；
- 进行 bulk RDMA；
- Decode 侧再 scatter 到本地 KV pool；
- Decode 使用预分配 ring buffer allocator；
- buffer 可以来自普通 cudaMalloc 或 backend custom memory pool。

来源：`python/sglang/srt/disaggregation/common/staging_buffer.py:1-11,117-263`、`staging_handler.py:30-129`。

源码仍保留“等 Triton kernels 在生产配置验证后移除 Torch fallback”的 TODO，因此此路径在产品能力矩阵中应按 connector、P/D topology 和目标设备标记 `experimental` 或 `verified`，不能因为代码存在就默认支持。

staging 是显式 GPU 常驻项，大小由配置/环境和 connector 初始化决定；它不应混入理论 KV payload，而应单独进入 P/D 相应 worker 的 runtime/connector 账本。

### 13.5 PD 与并行组合

PD 不改变前述 TP/PP/DP/EP 基本语义，但增加角色限制和 connector兼容性。例如：

- prefill CP 在 PD 中只允许 prefill 侧；decode-only 侧显式拒绝；
- P/D 可以使用不同 topology，但 KV layout/TP mapping 和 staging 是否支持需要 connector证据；
- DP Attention registered tests 包含 disaggregation 场景，但不能外推到所有模型/backend；
- 同模型/checkpoint和同框架版本仍是规划器已确认的顶层约束。

## 14. 模型和 backend 能力判定

### 14.1 为什么不能只维护一个模型白名单

DP Attention CLI help 的模型枚举落后于 exact tag 的模型接入范围；同时，部分模型虽然调用通用 helper，却可能在 lm_head、speculative、PP、量化、attention backend 或 A2A 上有额外限制。

例如：

- MiMo V2 的 fused QKV checkpoint 要求特定 effective Attention-TP；源码会根据 `TP/DP/CP` 组合拒绝不匹配配置。见 `python/sglang/srt/server_args.py:4058-4082`。
- Triton kernel MoE 的某些路径要求 `EP == 1`。见 `python/sglang/srt/server_args.py:4053-4056`。
- FlashInfer MoE A2A 要求 DP Attention 且 `DP == TP`。见 `python/sglang/srt/server_args.py:5142-5143`。
- DSA prefill CP 会改变 Graph/backend 行为。见 `python/sglang/srt/server_args.py:3895-3937`。

所以能力键至少是：

```text
framework version
+ model architecture/checkpoint layout
+ device platform
+ attention backend
+ quantization/load format
+ TP/PP/DP/Attention-CP/DCP/EP/MoE-DP
+ speculative mode
+ serving or PD role
+ connector/A2A backend
```

### 14.2 exact-tag 测试证据

v0.5.15 中存在以下注册测试目录，可作为组合 evidence 的来源：

- `test/registered/dp_attn/`
- `test/registered/disaggregation/test_disaggregation_dp_attention.py`
- `test/registered/spec/eagle/test_eagle_dp_attention.py`
- `test/registered/spec/eagle/test_eagle_infer_beta_dp_attention.py`
- `test/registered/unit/distributed/test_parallel_state.py`
- `test/registered/unit/managers/test_data_parallel_controller.py`

但目录存在不等于本次已在 H20 实跑通过。本审计只确认源码和测试资产存在；产品 `verified` 还需要记录具体 test case、模型、硬件、镜像和执行结果。

### 14.3 建议的证据状态

后续 adapter 对每个内置模型组合返回：

- `verified`：同时有 exact-version 组合证据和本项目目标平台实测；
- `implementation-capable`：组件代码已接入，但目标组合缺验证；
- `experimental`：依赖环境开关、fallback 或源码 TODO；
- `unsupported`：明确冲突或缺少必要组件。

未达到 `verified` 不代表一定 OOM；它表示 planner 不能给出“保证可启动”的显存结论。

## 16. 对显存规划器的设计输入

本次源码审计可以确认以下事实，但不在本文决定最终接口形式。

### 16.1 adapter 必须解析到物理 worker

正确顺序应为：

```text
用户输入/命令
  -> SGLang v0.5.15 默认值和自动改写
  -> 普通 DP 或 DP Attention worker 公式
  -> pp_rank/tp_rank 和物理 GPU 放置
  -> Attention 与 MoE component groups
  -> scheduler/cache partition
  -> 每张物理 GPU 的权重、cache 和运行时项
```

### 16.2 理论可确定部分

在固定 checkpoint tensor metadata、模型 adapter 和 resolved topology 后，可以较确定地计算：

- 每个 PP stage 的 checkpoint tensor 放置；
- Column/Row/QKV/vocab shard；
- GQA/MQA KV head复制；
- routed expert、shared expert、dense MLP 的权重归属；
- 标准 KV、MLA latent、DSA indexer、MiniMax indexer、SWA、state cache payload；
- page alignment 后的理论 cache bytes/token。

### 16.3 需要预算或校准部分

以下项不能仅靠 checkpoint 理论公式保证：

- prefill/decode activation峰值；
- DP gather/scatter global buffer；
- Attention-CP/DCP workspace；
- MoE A2A dispatch/combine buffer；
- CUDA Graph capture pool；
- allocator fragmentation；
- NCCL/通信库和 backend context；
- quantization repack/load peak；
- PD connector注册、metadata 和 staging；
- speculative/draft runtime临时对象。

### 16.4 结论必须保留证据边界

当模型/topology/backend组合没有 exact-version adapter 或验证证据时，应返回 `unsupported` 或“不足以保证”的状态，而不是套用 dense TP 通用公式。只有理论下界已超过物理显存时才能确定 OOM；只有保守上界和所有必要 runtime 项都有适用证据时才能保证可启动。
