# vLLM v0.24.0 并行策略源码审计

> 状态：方案设计输入，非产品实现规范
>
> 审计日期：2026-07-12
>
> 目标版本：vLLM `v0.24.0`
>
> 固定 commit：`ee0da84ab9e04ac7610e28580af62c365e898389`
>
> 目的：从目标版本源码确认并行参数如何变成实际 worker、通信组、模型组件放置、KV cache 所有权和显存占用，避免按参数字面含义推导显存

## 1. 执行摘要

vLLM 的 TP、PP、DP、EP、DCP 和 PCP 不是六种可以独立计算后相加的 rank。一个 GPU worker 同时拥有多种并行坐标；显存由该物理 worker 实际加载的模型组件、管理的 cache、捕获的 Graph 和运行时 workspace 共同决定。

本次审计确认：

1. 单个 DP engine 内的模型 worker 数为 `PP × PCP × TP`。整个非 ExternalDP 部署的模型 worker 总数为 `DP × PP × PCP × TP`。DCP 复用 TP worker，不增加 GPU。
2. rank 布局按 `ExternalDP × DP × PP × PCP × TP` reshape；运行时分别从同一 rank 张量建立 TP、DCP、PCP、PP、DP、EP 和 EPLB group。
3. TP 不是“总权重除以 TP”。Column/Row linear、QKV、vocabulary、norm/bias、量化辅助 tensor 的分片或复制规则不同。GQA/MQA 在 `TP >= KV head 数` 后复制 KV head，继续增加 TP 不再降低每 rank 的 KV 投影和 KV cache。
4. PP 按 layer 切分，而不是平均切总 checkpoint 字节。首尾 stage 还分别持有 embedding、final norm、lm_head 等组件；层数不整除时采用特定余数分配算法。KV 容量最终统一为所有 worker 中的最小 block 数。
5. Dense DP rank 是彼此独立的 EngineCore、scheduler、权重副本和 KV cache。MoE DP 则会让 expert 层跨 `DP × PCP × TP` 域协同，因此不能用同一套“独立副本”解释全部模型。
6. `--enable-expert-parallel` 改变 routed expert 权重放置：关闭时 expert tensor 在展平域做 tensor sharding；开启时每个 rank 持有完整的本地 expert 子集。EPLB 还会引入冗余 expert和独立通信组，并可能产生需要按 communicator/实现校准的迁移相关峰值。
7. DCP 已有明确的 decode KV token 维切分语义，但有 attention backend、cache 类型和 speculative 组合约束。PCP 在 v0.24.0 中仍是开发中能力：接口和 group 已存在，但内置 attention implementation 没有声明支持，MVP 不应将其视作可运行配置。
8. activation、CUDA Graph、allocator、通信和 kernel workspace 不是只由并行度闭式决定。v0.24.0 自身依赖 profile/capture 测量；规划器应使用版本化预算或校准区间，而不是伪精确公式。

## 3. 从配置到物理 worker

### 3.1 公开并行参数及重要默认值

`ParallelConfig` 的核心字段包括：

| 配置 | 默认值 | 运行时含义 |
|---|---:|---|
| `pipeline_parallel_size` | 1 | transformer layer stage 数 |
| `tensor_parallel_size` | 1 | 一个 stage 内的 tensor parallel 宽度 |
| `prefill_context_parallel_size` | 1 | PCP 宽度，会进入 worker world size |
| `data_parallel_size` | 1 | DP engine 数；MoE expert 域也使用它 |
| `decode_context_parallel_size` | 1 | DCP 宽度，复用 TP worker |
| `enable_expert_parallel` | false | 将 MoE routed experts 从 tensor sharding 改为 expert placement |
| `enable_eplb` | false | 启用 expert load balancing；要求 EP 已启用 |
| `all2all_backend` | `allgather_reducescatter` | MoE token dispatch/combine backend |
| `distributed_executor_backend` | 自动 | 单机通常 `mp`，多机可为 `mp`/Ray/外部 launcher |

字段定义及默认值见 `vllm/config/parallel.py:116-206,324-355`；CLI 暴露位置见 `vllm/engine/arg_utils.py:961-1117`。

非 `external_launcher` 路径下，两个容易误读的派生量是：

```text
parallel_config.world_size = PP × TP × PCP
parallel_config.world_size_across_dp = DP × PP × TP × PCP
```

构造逻辑见 `vllm/config/parallel.py:791-802`，`world_size_across_dp` 属性见 `vllm/config/parallel.py:516-520`。源码属性 docstring 仍写 `TP×PP` 或 `TP×PP×DP`，但普通路径的实际表达式已经包含 PCP。

`external_launcher` 是例外：初始化会先将 DP 乘入 `world_size`，而 `world_size_across_dp` 属性若直接读取仍会再次乘 DP。无论这是调用约定还是目标版本的实现边界，planner 都不能用该派生属性直接推导 external launcher 的物理 worker 数；必须按 launcher 提供的 distributed world 和 rank 语义专项解析。MVP 若没有对应验证，应将该路径标为 `unverified/unsupported`。

DCP 不进入上述乘积。配置验证要求 `TP % DCP == 0`，并明确说明 DCP 复用 TP GPU，见 `vllm/config/parallel.py:498-511`。

### 3.2 rank 坐标不是多个独立进程

`initialize_model_parallel` 将全局 rank reshape 为：

```text
ExternalDP × DP × PP × PCP × TP
```

其中 TP 是最内层、变化最快的坐标。注释和 reshape 见 `vllm/distributed/parallel_state.py:1760-1775`。

因此一个物理 worker 可以同时表示：

```text
(dp_rank, pp_rank, pcp_rank, tp_rank, dcp_rank, ep_rank)
```

这些是同一 worker 在不同 group 中的坐标，不能计算成：

```text
TP rank 显存 + PP rank 显存 + DP rank 显存 + EP rank 显存
```

正确问题是：“给定物理 worker 的组合坐标，它实际驻留了哪些组件和 buffer？”

### 3.3 实际建立的通信组

同一个 `all_ranks` 张量被转换成多组视图：

| group | 形成方式 | 主要作用 |
|---|---|---|
| TP | 每个连续 TP rank 一组 | dense/attention tensor collective |
| DCP | 将 TP 内连续 rank 再按 DCP 分组 | decode KV token 维协同 |
| PCP | 转置 PCP/TP 后按 PCP 分组 | prefill context 协同 |
| PP | 转置 PP/TP 后按 PP 分组 | stage 间 hidden state 传递 |
| DP | 转置 DP/TP 后按 DP 分组 | MoE DP 同步及状态协调 |
| EP | 每个 PP stage 内展平 `DP × PCP × TP` | routed expert dispatch/placement |
| EPLB | 与 EP 相同的 ranks，独立 process group | expert 负载统计和权重迁移隔离 |

构造代码分别位于 `vllm/distributed/parallel_state.py:1777-1916`。EP 仅在模型是 MoE 或模型配置尚未建立时创建；EP 域明确为 `DP × PCP × TP`，见同文件 `1870-1896`。

ExternalDP 是模型 world 外部的独立生成域，注释明确说每个 external DP rank 可独立生成；普通 serving 设计不应把它混入用户配置的 model DP，见 `vllm/distributed/parallel_state.py:1760-1766`。

## 4. Tensor Parallel 的组件级权重放置

### 4.1 Column 和 Row linear

vLLM 的 dense transformer 层沿用 Megatron 风格 TP：

- `ColumnParallelLinear` 沿权重输出维切分，当前 rank 的输出宽度为 `output_size / TP`；bias 也按输出维分片。见 `vllm/model_executor/layers/linear.py:394-488`。
- `RowParallelLinear` 沿权重输入维切分，本 rank 的输入宽度为 `input_size / TP`，输出宽度保持完整；bias 是完整输出维 tensor，只在 rank 0 融入 GEMM，通常对 partial output 做 all-reduce。见 `vllm/model_executor/layers/linear.py:1493-1595,1628-1649`。

所以即使两个矩阵的主权重都接近 `1/TP`，bias、输出 buffer 和 collective workspace 也不相同。

### 4.2 Q、K、V 并不总是等比例分片

`QKVParallelLinear` 按 query heads 和 KV heads 分别处理：

```text
local query heads = total query heads / TP

if TP >= total KV heads:
    local KV heads = 1
    KV head replicas = TP / total KV heads
else:
    local KV heads = total KV heads / TP
    KV head replicas = 1
```

实现见 `vllm/model_executor/layers/linear.py:914-984`。加载 K/V shard 时使用 `tp_rank // num_kv_head_replicas`，证明多个 TP rank 会加载同一个 KV head，见同文件 `1278-1286`。

直接后果：

- MHA 在 head 数可整除时通常随 TP 缩小 Q/K/V；
- GQA 在 TP 未超过 KV head 数时仍可缩小 K/V；
- 当 TP 达到或超过 KV head 数后，每 rank 至少保留一个 KV head，K/V 权重不再继续下降；
- 对 MLA 或模型专用 projection，不能仅套用标准 GQA 公式，必须由模型 adapter 给出 tensor placement。

MiniMax-M3 的专用 loader 还显式将 `index_k` 复制到每个 rank，而其他 shard 使用 KV replica rank，见 `vllm/model_executor/layers/linear.py:1460-1488`。这说明即使同属一个 fused projection，indexer 组件也可能有独立放置规则。

### 4.3 Embedding 和 lm_head

`VocabParallelEmbedding` 按 padded vocabulary 维切分，每 rank 的行数为 `padded_vocab_size / TP`；forward 后对 embedding result 做 TP all-reduce。见 `vllm/model_executor/layers/vocab_parallel_embedding.py:250-325,472-492`。

`ParallelLMHead` 继承相同 vocabulary sharding，并可与 input embedding 绑定权重，见同文件 `503-560`。因此 planner 需要区分：

- vocabulary padding 后的驻留字节；
- embedding 与 lm_head 是否真正 tied；
- PP 首尾 stage 是否分别持有，还是模型 loader 共享了同一 parameter；
- quantization 是否支持 embedding/head，并产生额外 scales。

### 4.4 复制参数和量化辅助 tensor

TP 不会自动处理所有 parameter：

- Row parallel bias 保持完整输出维；
- norm 权重通常在各 TP rank 复制；
- 没有 `output_dim`/`input_dim` placement 元数据的 tensor 可能在所有 rank 原样复制；vocabulary loader 对此有明确分支，见 `vllm/model_executor/layers/vocab_parallel_embedding.py:430-445`；
- GPTQ/AWQ/FP8/bitsandbytes 等可能包含 packed weight、scale、zero point、codebook 或 `g_idx`，其分片需要 quant method 和 weight loader 决定；packed/block scale 的 offset 调整可见 `vllm/model_executor/layers/linear.py:1240-1278`。

因此权重计算应从 checkpoint tensor 和目标模型/量化 loader 的组件规则出发，而不是用“参数量 × 位宽 / TP”。

## 5. Pipeline Parallel：按 layer 和首尾组件放置

### 5.1 layer 划分算法

模型通过 `make_layers` 只实例化本 PP stage 的真实 transformer layers，其余位置使用 `PPMissingLayer` 占位，见 `vllm/model_executor/models/utils.py:627-672`。

默认 layer partition 不是简单的小数平均：

- 基础值为 `num_hidden_layers // PP`；
- 余数优先分配到除最后 stage 之外的 stage；
- `PP > 2` 且余数较少时倾向中间 stage，以避开同时承载 input/output embedding 的首尾 stage；
- 可由 `VLLM_PP_LAYER_PARTITION` 显式覆盖。

规则见 `vllm/distributed/utils.py:109-145`。

### 5.2 首尾 stage 具有额外权重

以 Llama 实现为例：

- 首 stage 负责输入 embedding 和从 `input_ids` 构造 hidden state；
- 非首 stage 接收 `IntermediateTensors`；
- 末 stage 才实例化 final RMSNorm 和 lm_head；
- 非末 stage 返回 PP intermediate tensors。

执行证据见 `vllm/model_executor/models/llama.py:380-431,486-520`。其他模型可能还有 vision encoder、MTP head、pooler 或模型专用 indexer；这些必须由模型 adapter 确定，不能从总 checkpoint 平均推断。

### 5.3 PP 的显存与容量聚合规则

每个 PP worker 只为自己的 layers 注册 KV spec。中央 KV controller 会：

1. 合并全模型的 cache specs；
2. 将 global cache group 投影回每个 worker 的实际 layer；
3. 按每 worker 的可用显存做 max context/admission 检查；
4. 生成各 worker cache config；
5. 将所有 worker 的 `num_blocks` 收敛为最小值。

源码说明和实现见 `vllm/v1/core/kv_cache_utils.py:2010-2134`。

这意味着：

- “服务 cache 容量”不是各 PP rank cache token 的和；
- 容量由最紧张 PP/TP/PCP worker 决定；
- 不同 stage layer 数、额外首尾权重、Graph/activation 和 runtime 差异都会改变可用 KV bytes；
- planner 必须先计算每 worker，再取限制 worker，不能把总卡显存汇总后比较。

## 6. Data Parallel 的两种内部语义

### 6.1 Dense 模型：完全独立副本

启动 DP EngineCore 时，vLLM 对非 MoE 模型将内部 `data_parallel_size` 重置为 1，并注明“Non-MoE DP ranks are completely independent”。原始 DP index 只用于标识和路由，见 `vllm/v1/engine/core.py:1153-1200`。

因此 Dense DP rank 具有：

- 完整的 `PP × PCP × TP` 模型副本；
- 独立 EngineCore 和 scheduler；
- 独立 KV cache block pool；
- 独立 CUDA Graph/runtime 状态；
- 一个请求只被路由至一个 DP engine，而不是在 DP rank 间切分。

内部负载均衡客户端依据每个 EngineCore 的 waiting/running request 计数选择目标，评分为 `waiting × 4 + running`，见 `vllm/v1/engine/core_client.py:1380-1441`。该评分不考虑 prefix cache 命中和每个 engine 的实际 KV bytes，因此显存规划不能假设运行请求会完美均匀分布。

### 6.2 MoE 模型：attention DP，expert 跨 DP 协同

MoE DP rank 保留真实 `data_parallel_rank` 并使用 `DPEngineCoreProc`，见 `vllm/v1/engine/core.py:1188-1193`。DP ranks 仍各有请求和 scheduler，但 expert collective 要求各 rank 同步推进；没有可执行请求的 rank 会执行 dummy batch，避免 collective deadlock，见同文件 `1925-1961`。

这对应 vLLM 文档所称的 Data Parallel Attention：attention、KV 和非 expert 组件可按 DP rank 复制并处理不同请求，而 MoE expert 层在更宽域上交换 token。它不是一个额外布尔参数，也不能简化成“整个模型复制 DP 次”。

### 6.3 对服务容量的含义

- Dense DP 的整体 token 容量是多个独立 scheduler 分区容量的集合；单请求不能跨 DP 分区使用 KV。
- MoE DP 也具有 per-rank scheduler/KV ownership，但执行波次受跨 DP expert collective 协调。
- 对“整个服务的最大 token”，可以汇总各 DP 分区的可用 token；同时必须单独报告单分区容量和单请求最大上下文。
- 最坏显存判断仍逐物理 worker 进行，不能用 DP aggregate memory 抵消单卡不足。

## 7. MoE：Tensor Parallel、Expert Parallel 与 EPLB

### 7.1 effective expert domain

Fused MoE 会先把原始并行维度展平：

```text
expert domain size = DP × PCP × TP
flatten rank = dp_rank × PCP × TP + pcp_rank × TP + tp_rank
```

实现见 `vllm/model_executor/layers/fused_moe/config.py:1098-1107`。PP 不进入该乘积，因为每个 PP stage 只处理自己的 MoE layers。

### 7.2 EP 关闭与开启的权重差异

当 `enable_expert_parallel=false` 时：

- `ep_size=1`；
- Fused MoE 将展平域作为 effective TP；
- expert tensor 在 `DP × PCP × TP` rank 上做 tensor sharding。

当 `enable_expert_parallel=true` 时：

- effective MoE `tp_size=1`；
- `ep_size=DP × PCP × TP`；
- 每个 rank 持有完整的本地 expert 子集，token 通过 all-to-all/dispatch 到 owner rank。

选择和返回配置见 `vllm/model_executor/layers/fused_moe/config.py:1190-1237`。所以 `enable_expert_parallel` 会实质改变 resident weight bytes、量化 scales 形状、通信 buffer 和 activation routing；它不是只影响性能的开关。

### 7.3 不能把全部 MoE 参数等分为 EP

EP 通常只作用于 routed experts。以下组件需要独立规则：

- attention 和 norm；
- router/gate；
- shared experts；
- dense layer 或非 routed FFN；
- expert bias、quant scales 和 metadata；
- MTP/speculative 模块；
- vocabulary embedding/lm_head。

具体模型可能覆写 expert loader 或采用 fused 3D checkpoint。规划器必须按模型组件记录 placement，并将未经适配的模型/量化组合判为不支持，而不是对 checkpoint 中所有 `expert` 名称机械除以 EP。

### 7.4 EPLB 的额外驻留和峰值

EPLB 要求：

- `enable_expert_parallel=true`；
- `TP × DP > 1`；
- CUDA-like 平台；
- 冗余 expert 数只能在 EPLB 开启时非零。

验证见 `vllm/config/parallel.py:475-495`。EPLB 使用与 EP 相同 ranks 但独立的 process group，避免负载均衡通信与 MoE forward collective 互相死锁，见 `vllm/distributed/parallel_state.py:1898-1916`。

显存规划至少要考虑：

- `num_redundant_experts` 带来的额外 resident expert 权重；
- load statistics 和 placement metadata；
- expert weight 迁移期间 source、destination 和 staging buffer 是否并存；
- communicator 不同产生的 CPU/GPU staging 差异。

目标版本会在 NIXL、PyNccl、Gloo 等 communicator 间自动选择，逻辑见 `vllm/config/parallel.py:933-948`。这些峰值不能仅凭 expert 最终布局精确计算，应进入 framework/version calibration 或显式预算。

## 8. Decode Context Parallel（DCP）

### 8.1 DCP 不增加 GPU

DCP 把每个 TP group 切成若干大小为 DCP 的 group；`TP % DCP == 0`。parallel state 直接对现有 rank reshape，没有将 DCP 乘入 world size，见 `vllm/distributed/parallel_state.py:1794-1814`。

它的目标是沿 token/context 维进一步切分 decode KV，尤其用于 `TP > KV heads` 后 KV head 已复制的情况。目标版本文档给出的约束范围是：

```text
1 <= DCP <= TP / KV_heads
```

并说明 DCP 只降低 KV 复制，不增加 worker，见 `docs/serving/context_parallel_deployment.md:19-43`。

### 8.2 每 rank KV memory

通用 attention cache spec 在估算最大 context 内存时使用：

```text
local max tokens = ceil(max_model_len / (DCP × PCP))
local bytes = ceil(local max tokens / block_size) × page_size_bytes
```

实现见 `vllm/v1/kv_cache_interface.py:232-244`。这里 `page_size_bytes` 已由本 rank 的 local KV head/latent 布局决定，因此 planner 需要同时处理：

1. TP 对 head/latent 维的分片或复制；
2. DCP 对 token 维的切分；
3. block/page 向上取整；
4. PP stage 的 cache layer 数。

### 8.3 支持约束

DCP 不是对任意 cache/backend 的无条件公式变换：

- attention implementation 必须在 decode 返回 softmax LSE，否则初始化 assert 失败，见 `vllm/v1/worker/cp_utils.py:14-44`；
- `a2a` DCP backend 要求 `DCP > 1`，见 `vllm/config/parallel.py:509-512`；
- MTP 与非平凡 cache interleave 需要 implementation 明确支持；
- 部分 sliding-window、hybrid attention、Mamba/state cache 或模型专用 cache spec 有不同限制，不能套用标准 attention KV 公式；
- 通信 backend 会引入 all-gather/reduce-scatter 或 all-to-all 的 LSE/output workspace。

因此 planner 只能对“模型 adapter + attention/cache backend capability matrix”中已确认的 DCP 组合计算；其他组合应阻止可执行导出。

## 9. Prefill Context Parallel（PCP）

### 9.1 代码骨架已经进入拓扑

PCP 与 DCP 不同：

- PCP 乘入 `world_size`，会增加 worker；
- PCP 建立独立 process group；
- PCP 参与 MoE expert domain；
- cache spec 将 local context 按 `DCP × PCP` 缩小。

证据分别见 `vllm/config/parallel.py:793-797`、`vllm/distributed/parallel_state.py:1816-1833`、`vllm/model_executor/layers/fused_moe/config.py:1098-1107`、`vllm/v1/kv_cache_interface.py:236-244`。

### 9.2 v0.24.0 不应作为正式可运行能力

目标版本官方文档明确称 PCP 的 partial-query/full-KV 和 partial-query/partial-KV 两种策略仍在 active development，见 `docs/serving/context_parallel_deployment.md:8-17`。

更关键的是，attention implementation 基类的 `supports_pcp` 默认是 `False`，见 `vllm/v1/attention/backend.py:734-755`；对 v0.24.0 源码的全仓搜索只找到该默认定义和运行时检查，没有内置 implementation 将它设为 `True`。当 `PCP > 1` 时，兼容性检查会 assert `layer_impl.supports_pcp`，见 `vllm/v1/worker/cp_utils.py:39-44`。

结论：

- v0.24.0 中 PCP 参数、rank group 和部分 cache 语义存在；
- upstream 内置 CUDA attention path 没有形成可依赖的完整支持；
- MVP 应将 `PCP > 1` 标记为 `unsupported/experimental`，不能输出“可运行”；
- 未来升级版本时必须重新基于 exact release tag 审计，不能沿用本结论或仅检测 CLI 参数存在。

## 10. 请求路由、scheduler 与 cache ownership

### 10.1 一个请求如何落到 DP engine

内部 DP LB 为每个 EngineCore 维护 waiting/running 数量并选择分数最小的 engine。请求发送后会记录 engine identity，以便 abort 回到原 engine，见 `vllm/v1/engine/core_client.py:1413-1441`。

请求进入目标 EngineCore 后，由该 engine 的 scheduler 管理 block table 和 KV cache。没有逻辑把一个普通请求的 KV 自动分摊到其他独立 DP engine。DCP/PCP 是 model worker 内部的 context parallel，不能和 DP LB 混为一谈。

### 10.2 MoE DP 的同步执行波次

MoE DP ranks 可以收到不同请求，但 expert collective 要求 group 内 ranks 同步进入 forward。`DPEngineCoreProc` 会通过 DP all-reduce 判断全局是否仍有 unfinished requests，并在本地无 ready request 时执行 dummy batch，见 `vllm/v1/engine/core.py:1863-1875,1925-1961`。

这会产生两类显存影响：

- 空闲 rank 仍需保留正常模型、Graph 和 collective workspace；
- dummy batch 的 activation 峰值可能小于真实 batch，但不能假定空闲 rank 的运行时显存为零。

### 10.3 prefix cache 与容量不能假设均衡

LB 分数只看请求计数，不看请求 token 数、已占 KV blocks 或 prefix affinity。因此同一时刻各 DP engine 的 KV 使用可明显不均。规划器若要保证目标服务启动和上限配置不 OOM，应使用每分区配置上界，而不是假定 aggregate token 被 DP 完美平均。

## 11. 多机启动与 rank 放置

### 11.1 framework 的常规建议

vLLM 官方建议：

- 单机模型并行优先使用 TP；
- 多机时通常令 `TP = 每机 GPU 数`、`PP = 节点数`；
- rank 连续编号，尽量让相邻 TP ranks 位于同一高速互联节点。

见 `docs/serving/parallelism_scaling.md:3-33,115-159`，parallel state 也明确提醒调用者让相邻 ranks 位于同一 DGX，见 `vllm/distributed/parallel_state.py:1711-1722`。

这是一种推荐 placement，不是所有 topology 的硬编码等式。Ray 也允许 TP 跨节点，但会改变通信路径和 workspace/runtime 风险。

### 11.2 multiprocessing 的自动 rank

多节点 MP executor 按节点内连续 local ranks 生成 global rank：

```text
global_start_rank = local_world_size × node_rank_within_dp
global_rank = global_start_rank + local_rank
```

见 `vllm/v1/executor/multiproc_executor.py:120-190`。`local_world_size = world_size / nnodes_within_dp`，并要求整除，见同文件 `249-260` 及 `vllm/config/parallel.py:671-685`。

`ParallelConfig` 在单机可容纳 world 时默认使用 `mp`，显式多机 CUDA 也选择 `mp`；若本机 GPU 数不足且未配置 Ray 或 `nnodes`，会报错而不是静默跨机，见 `vllm/config/parallel.py:871-916`。

### 11.3 规划器需要解析的放置信息

为了重建物理 GPU 账本，vLLM adapter 至少需要：

- `TP/PP/DP/PCP/DCP`；
- `nnodes`、每节点 GPU 数、`node_rank` 或 Ray placement；
- DP local size/start rank/external or hybrid LB；
- distributed executor backend；
- framework 自动产生的连续 global/local rank 映射；
- 每个 rank 的 PP layer interval 和模型组件清单。

MVP 不需要预测通信性能，但必须知道 collective 是否跨节点，因为 backend capability、Graph/runtime 校准和可执行命令兼容性会受影响。

## 12. 显存分项如何受并行策略影响

### 12.1 可以从模型与放置确定的分项

在模型/checkpoint 和 adapter 固定时，可高确定性计算：

| 分项 | 必要放置信息 |
|---|---|
| attention Q/K/V/O 权重 | PP layers、TP head/row/column shard、KV head replication |
| dense FFN 权重 | PP layers、column/row TP |
| routed experts | PP layers、effective MoE TP 或 EP owner、冗余 expert |
| shared experts/router | 模型专用 placement，不随 routed EP 机械等分 |
| embedding/lm_head | PP 首尾 stage、vocab TP、padding、tie 状态 |
| norm/bias/scales | 逐 tensor 分片或复制元数据 |
| KV/latent/indexer cache | cache spec、local heads/latent、PP layers、DCP/PCP、block rounding |
| Mamba/linear state | state cache 专用 spec，不能当标准 KV |

### 12.2 需要版本化校准或预算的分项

v0.24.0 自己通过 profile run 测量 weight、peak activation 和 non-Torch memory，再从 `gpu_memory_utilization` 预算中扣除这些项得到 available KV bytes，见 `vllm/v1/worker/gpu_worker.py:400-510`。

CUDA Graph 更不能由“capture size × 固定系数”可靠推导。目标版本会：

- 枚举实际 capture descriptors；
- 对每种 mode 捕获样本并测 free memory 差值；
- 将 FULL/PIECEWISE 的共享 pool 按 max overlay，而不是相加；
- 将独立 encoder graph pool 额外相加；
- 真正 capture 后再次测量总占用。

实现见 `vllm/v1/worker/gpu_model_runner.py:6446-6593,6595-6661`。

规划器应将以下内容建模为 framework/version/hardware/model/topology 条件下的校准区间或用户预算：

- peak activation 和 chunked prefill 临时 tensor；
- CUDA Graph pool、graph exec 和静态 input/output buffer；
- Torch allocator reserved/fragmentation；
- NCCL/custom all-reduce/PP/EP/DCP workspace；
- quant kernel workspace；
- EPLB migration staging；
- compilation、warmup 和 Graph capture 初始化峰值。

### 12.3 初始化峰值与稳态峰值必须分开

服务“起得来”要求初始化阶段也不 OOM。至少应区分：

1. checkpoint loading/quant preparation；
2. profile run；
3. CUDA Graph capture；
4. Prefill 稳态；
5. Decode 稳态；
6. 可选 EPLB/connector 操作。

目标版本对显存 profiler 轻微低估还保留 150 MiB redundancy buffer，见 `vllm/v1/worker/gpu_worker.py:680-728`。这不能替代产品统一安全余量，但证明 framework 本身也不把实测 profile 当作无误差点值。

## 13. 能力矩阵和 MVP 判定建议

以下是基于 v0.24.0 源码可以形成的能力状态，不是对任意模型组合的运行认证：

| 能力 | 状态 | 规划器处理 |
|---|---|---|
| Dense TP | 源码确认 | 模型 adapter 给出逐组件 TP placement |
| Dense PP | 源码确认 | 解析实际 layer partition 和首尾组件，取最坏 worker |
| Dense DP | 源码确认 | 独立副本、scheduler、KV 分区；整体与单分区容量分开 |
| MoE DP + MoE tensor sharding | 条件支持 | effective domain 为 `DP × TP`；MVP PCP 固定 1 |
| MoE EP | 条件支持 | 仅对已适配模型、量化和 A2A backend 开放 |
| EPLB | 条件支持 | 需额外权重/迁移预算；无校准时结论为“可能”或不支持 |
| DCP | 条件支持 | capability matrix 校验 attention/cache/backend/speculative 组合 |
| PCP | 不支持/实验性 | v0.24.0 MVP 阻止计算和可执行导出 |
| Elastic EP | 高复杂度条件支持 | 若 MVP 无明确需求，建议不支持而非近似 |
| DBO/ubatching | 条件支持 | 会改 activation/Graph/runtime；无适用校准不输出确定“可运行” |

首批模型不能只登记“vLLM 支持该 architecture”，而应登记到至少以下粒度：

```text
model checkpoint
× quantization/load format
× attention/cache backend
× TP/PP/DP/EP/DCP topology constraints
× CUDA Graph mode
× PD role/connector（如使用）
```

不在能力矩阵的组合应明确输出 `unsupported` 或 `unverified`，不能用理论平均值继续生成可执行命令。

## 14. 对 LLM Memory Planner 的设计输入

本次审计支持以下设计事实，但不在本文确定最终 schema。

### 14.1 framework adapter 必须先解析拓扑

vLLM adapter 应完成：

```text
用户/命令参数
  -> v0.24.0 默认值、校验和自动选择
  -> EngineCore、worker、rank 坐标和 process groups
  -> PP layer partition
  -> 模型组件 placement
  -> scheduler/KV cache 分区
  -> 每物理 GPU 显存账本输入
```

不能让共享计算核心自行猜测 vLLM 的 rank 或 group 语义。

### 14.2 共享层可以接收组件归属，不接收模糊并行除数

共享显存账本可以处理已经解析后的事实，例如：

- GPU 0 拥有 layer 0-15；
- Q shard 为 1/8，KV head 0 在 TP rank 0/1 复制；
- routed expert 0-7 归该 EP rank，另有 1 个 EPLB redundant expert；
- 本 worker 的 KV page bytes、block 数和 cache groups；
- Graph/activation/runtime 的预算区间。

共享层不应接收“权重共 400 GB，TP=8，EP=8”后自行平均分配。

### 14.3 输出必须定位限制 worker

结果至少需要解释：

- 限制节点和物理 GPU；
- `dp/pp/pcp/tp/dcp/ep` 组合坐标；
- PP layer 范围；
- 哪个阶段达到峰值；
- 主要显存瓶颈是权重、cache、Graph、activation 还是 runtime；
- 服务整体 token 容量、每 scheduler 分区容量和单请求上下文上限。

### 14.4 理论项和经验项保持证据边界

- checkpoint tensor placement 和 cache payload 可形成确定性上下界；
- framework profile 才能观测的 Graph/activation/runtime 使用校准区间；
- 若区间跨越可用显存阈值，结论是“可能运行，需要实测”，不是伪精确点值；
- 若 lower bound 已超过物理显存，才判定确定 OOM；
- 若 conservative upper bound 连同安全余量都满足，才判定可运行。

## 16. 审计结论

vLLM v0.24.0 的显存规划边界应是“framework adapter 解析实际 worker 和组件放置，再交给共享显存账本”，而不是“为 TP/PP/DP/EP/CP 各写一个除法公式”。

可以稳定下沉到共享层的是物理 GPU、组件 resident bytes、cache payload、阶段和经验预算；必须保留在 vLLM adapter 中的是 rank/group 形成、PP layer partition、QKV replication、Dense/MoE DP 分流、MoE effective domain、DCP capability 和 PCP 版本状态。

本文只提供后续方案设计的证据输入。最终统一接口、模型 capability matrix 的具体结构，以及首批支持组合，应在 vLLM、SGLang 两份独立审计和框架对比文档全部审阅后再决定。
