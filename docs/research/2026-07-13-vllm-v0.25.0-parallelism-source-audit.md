# vLLM v0.25.0 并行策略源码审计

> 状态：设计输入，不是最终产品方案
> 目标版本：vLLM `v0.25.0`
> tag commit：`702f4814fe54fabff350d43cb753ae3e47c0c276`
> 审计日期：2026-07-13

如果还不熟悉大模型推理并行，建议先阅读[并行策略源码阅读基础知识](./2026-07-13-parallelism-source-reading-foundations.md)。该文档先解释物理 GPU、worker、rank、process group、collective、TP/PP/DP/EP/CP、KV page 和运行阶段；本文在此基础上只讨论这些概念在 vLLM `v0.25.0` 中的具体实现。

## 1. 结论摘要

vLLM 的 TP、PP、DP、EP、DCP 和 PCP 不是六种可以分别计算再相加的
“rank”。一张物理 GPU 上的一个 worker 会同时具有 TP/PP/DP/PCP 等坐标；
显存由这个 worker 实际拥有的权重、cache、持久 buffer、CUDA Graph 和阶段峰值
共同决定。

对显存规划器最重要的结论如下。

1. 常规单个 DP engine 的模型 worker 数是 `PP * PCP * TP`；整个部署通常是
   `DP * PP * PCP * TP`。DCP 复用 TP worker，不增加 GPU。
2. 当 model-parallel world 中包含非平凡 DP 轴时，进程组按
   `ExternalDP * DP * PP * PCP * TP` 的 rank 张量建立 TP、DCP、PCP、PP、DP、
   EP 和 EPLB group。普通非 MoE Dense DP 是多个独立 EngineCore worlds，每个
   world 内的 DP 轴重置为 1；部署层仍需在这些 worlds 外表达 DP 副本坐标。
3. TP 不能解释为“所有权重除以 TP”。Column/Row linear、vocab、Q/K/V、norm、
   bias、量化 scale 的放置规则不同；当 `TP > KV heads` 时，KV heads 会复制。
4. PP 按层切分，embedding、final norm 和 lm_head 只在特定 stage；最坏 stage
   不能用平均权重代替。所有 worker 的 KV block 数最终取最小值。
5. Dense DP 是独立权重副本、scheduler 和 KV cache；MoE DP 会让 expert 层跨
   `DP * PCP * TP` 域协同，并可能为 collective 执行 dummy forward。
6. `enable_expert_parallel` 会把 routed experts 从展平域上的 tensor sharding 改为
   expert placement。EPLB 的冗余专家、迁移 communicator 和 staging 另有显存。
7. DCP 已有 decode KV token 维切分语义，但必须按模型、attention backend、cache
   类型、MTP 和 connector 组合校验。
8. PCP 在 `v0.25.0` 仍不能作为正式可运行能力：配置、group 和部分 cache 代码
   已存在，但没有内置 attention implementation 声明 `supports_pcp=True`，官方
   文档仍称两种 PCP 路线处于开发中。
9. `v0.25.0` 更容易默认选择 Model Runner V2。V1 会在 KV 预算前估算 CUDA
   Graph 显存，但只有开启 `VLLM_MEMORY_PROFILER_ESTIMATE_CUDAGRAPHS` 才会预扣
   该估值；V2 的 `profile_cudagraph_memory()` 仍固定返回 0，只在真正 capture 时
   测量。因此 runner 版本属于显存语义，不能作为内部细节忽略。
10. `external_launcher` 的 `world_size` 字段具有特殊语义：初始化时已经乘入 DP，
    而 `world_size_across_dp` property 又乘一次 DP。后者不能在该模式下无条件当作
    实际进程总数；必须结合 launcher 和 torch distributed 的真实 world 解析。

## 3. 新人基础：process、rank、world 和 group

### 3.1 一个 rank 不是一张额外的卡

在典型 GPU 推理中，一个模型 worker process 绑定一张 GPU。`rank` 是这个 process
在某个通信集合中的编号。例如同一个物理 worker 可以同时是：

```text
global rank = 11
TP rank     = 3 / 8
PP rank     = 1 / 2
DP rank     = 0 / 2
DCP rank    = 3 / 8
```

这些不是五个 process，也不能把五份“rank 显存”相加。它们只是从不同角度描述
同一个 worker 的位置。

### 3.2 world size 和 process group

`world size` 是某个 distributed world 中的 process 数。`process group` 是 world 的
子集，决定 collective 在哪些 ranks 间发生：

- TP group：同一 PP/DP/PCP 坐标下切 tensor；
- PP group：同一 TP/DP/PCP 坐标下串联不同 layer stages；
- DP group：相同模型坐标的不同数据副本；
- DCP group：TP 内连续 ranks 的 decode-context 子组；
- PCP group：prefill context workers；
- EP group：同一 PP stage 内用于 routed experts 的宽域。

通信 group 本身也可能创建 NCCL/Gloo communicator 和 workspace，因此显存预算除
权重/cache 外还要包含 runtime/communication 项。

### 3.3 并行坐标为什么不能直接做除法

`TP=8` 只表示某些 tensor 沿某个维度分成八份。它不保证：

- 所有 tensor 都被切分；
- 八份没有 padding；
- KV heads 足以切八份；
- embedding、norm、bias、scale 与主 weight 使用同一规则；
- MoE expert 和 dense attention 使用同一 group。

规划器需要先解析“每个组件在哪些物理 worker 上、每份的真实 shape”，再记账。

## 4. 配置与 worker 数

`ParallelConfig` 的主要字段见 `vllm/config/parallel.py:116-230,324-372`：

| 字段 | 默认值 | 语义 |
|---|---:|---|
| `pipeline_parallel_size` | 1 | transformer stage 数 |
| `tensor_parallel_size` | 1 | stage 内 tensor sharding 宽度 |
| `prefill_context_parallel_size` | 1 | PCP 宽度，会增加 worker |
| `data_parallel_size` | 1 | DP engine 数，也是 MoE expert 域的一部分 |
| `decode_context_parallel_size` | 1 | DCP 宽度，复用 TP workers |
| `enable_expert_parallel` | false | routed expert placement 模式 |
| `enable_eplb` | false | expert load balance 及可选冗余专家 |
| `all2all_backend` | `allgather_reducescatter` | MoE dispatch/combine 实现 |
| `enable_dbo` | false | dual-batch overlap，可能增加并存 activation/workspace |

常规路径初始化：

```text
world_size           = PP * TP * PCP
world_size_across_dp = world_size * DP
```

源码：`vllm/config/parallel.py:791-801,516-520`。DCP 校验要求 `TP % DCP == 0`，
见 `vllm/config/parallel.py:498-512`。

### 4.1 external launcher 例外

`external_launcher` 在 `__post_init__` 中把 `world_size` 再乘 `DP`，并用
`RANK // (world_size / DP)` 推导 DP rank，见
`vllm/config/parallel.py:799-834`。与此同时 `world_size_across_dp` 仍返回
`world_size * DP`。

因此在这个模式中：

- `world_size` 已代表 launcher 的全局 TP/PP/PCP/DP processes；
- `world_size_across_dp` 会出现再次乘 DP 的字段级结果；
- planner 不能只读取 property 名称推导 GPU 数；
- 应以 launcher `WORLD_SIZE`、实际 rank reshape 和配置一致性共同解析。

external launcher 不让 vLLM 再扩展 DP world；`init_distributed_environment` 对该模式
跳过内部 world/rank offset，见 `vllm/distributed/parallel_state.py:1555-1605`。
CI 覆盖 TP、PP、DP+EP 的 torchrun 组合，见
`.buildkite/test_areas/distributed.yaml:116-127`。

## 5. Rank 布局和实际 process groups

`initialize_model_parallel` 在收到非平凡 DP 轴时使用的 rank tensor 是：

```text
ExternalDP * DP * PP * PCP * TP
```

对应源码 `vllm/distributed/parallel_state.py:1779-1794`。源码注释少列了 PCP，
应以紧邻的 reshape 为准；TP 是最快变化维度。普通非 MoE Dense DP 在每个
EngineCore 内先把 `data_parallel_size` 重置为 1，因此每个独立 world 中该张量的
DP 轴为 1，见 `vllm/v1/engine/core.py:1153-1200`。

group 构造如下：

| group | 形成方式 | 主要用途 |
|---|---|---|
| TP | 最后一维连续 ranks | tensor sharding/collective |
| DCP | 将 TP 连续切成 DCP 子组 | decode KV token sharding |
| PCP | 交换 PCP/TP 维后取 PCP | prefill context parallel |
| PP | 交换 PP/TP 维后取 PP | stage send/recv |
| DP | 交换 DP/TP 维后取 DP | MoE DP 同步/状态 |
| EP | 每个 PP stage 展平 `DP*PCP*TP` | expert dispatch/placement |
| EPLB | 和 EP 相同 ranks 的独立 group | 隔离 expert 迁移通信 |

源码：`vllm/distributed/parallel_state.py:1796-1938`。EP 只对 MoE 或模型配置尚未
建立时创建，见同文件 `1889-1915`。

## 6. TP 权重放置

### 6.1 Column 和 Row linear

- `ColumnParallelLinear` 沿 weight 输出维切分，local output size 为
  `output_size / TP`；可选 all-gather，见
  `vllm/model_executor/layers/linear.py:395-476,520-564`。
- `RowParallelLinear` 沿 weight 输入维切分，bias 不切分，结果通常 all-reduce，见
  `vllm/model_executor/layers/linear.py:1535-1629,1641-1695`。

所以 even dense transformer 权重也不能仅靠总参数量除 TP：bias、norm、部分
scale 和禁用 TP 的模块会复制，量化还会引入 packing、block scale 和 padding。

### 6.2 QKV 与 GQA/MQA 复制饱和

`QKVParallelLinear` 沿 attention heads 切分：

```text
local Q heads  = total Q heads / TP
if TP >= total KV heads:
    local KV heads    = 1
    KV replica factor = TP / total KV heads
else:
    local KV heads    = total KV heads / TP
```

源码：`vllm/model_executor/layers/linear.py:942-1008`。加载 K/V shard 时使用
`tp_rank // num_kv_head_replicas`，见同文件 `1298-1306`。

这意味着：

- Q 权重一般继续随 TP 下降；
- K/V 权重下降到每卡一个 KV head 后停止；
- 多个 TP ranks 持有相同 K/V shard；
- KV cache 也存在相同复制饱和，DCP 用 token 维切分来消除它。

`v0.25.0` 把 QKV `output_size` 改写成 `sum(output_sizes)`，并新增 loader/fuser
接口，但上述 head sharding 与复制语义没有改变。

### 6.3 vocab、embedding、lm_head

vocab embedding 按 padded vocabulary 切分，先分别 padding base/LoRA vocab，再在
TP ranks 间分配；不属于当前 rank 的 token 被 mask，输出 all-reduce。见
`vllm/model_executor/layers/vocab_parallel_embedding.py:196-270,430-492`。

显存公式必须计入 padded vocab，而不是只使用原始 `vocab_size`。

## 7. PP 权重与 KV 放置

PP 按 layer index 切分，未属于本 stage 的层使用无参数 `PPMissingLayer`。见
`vllm/model_executor/models/utils.py:672-717`。

以 Llama 为例：

- embedding 在第一 stage；tie embeddings 时末 stage 也可能持有；
- final norm 和 lm_head 在最后 stage；
- transformer layers 按 `get_pp_indices` 分配。

源码：`vllm/model_executor/models/llama.py:373-391,484-499`。默认分层会考虑首尾
stage 的 embedding/head 负担，也允许 `VLLM_PP_LAYER_PARTITION` 自定义，见
`vllm/distributed/utils.py:127-160`。

KV cache 按本 stage 实际 attention layers 配置。vLLM 会为各 worker 先按可用显存
生成配置，最后把所有 ranks 的 `num_blocks` 收敛到最小值并同比缩小 tensor，见
`vllm/v1/core/kv_cache_utils.py:2005-2028,2097-2153`。因此最紧张的 PP/TP worker
决定副本容量。

## 8. DP：Dense 与 MoE 必须分开

### 8.1 Dense DP

在线服务启动 engine core 时，非 MoE DP 会把每个 engine 内部配置重置成 DP=1，
但保留 `data_parallel_index` 用于路由。源码明确写着“Non-MoE DP ranks are
completely independent”，见 `vllm/v1/engine/core.py:1153-1200`。

每个 dense DP rank 因而拥有：

- 一套 `PP * PCP * TP` 模型 worker；
- 完整模型副本（内部仍按 TP/PP 切）；
- 独立 scheduler 和 KV cache；
- 独立 Graph/runtime budget。

服务总 token 容量可以聚合多个 DP partitions，但单请求不会跨普通 DP rank
共享 KV。

### 8.2 MoE DP

MoE DP 使用 `DPEngineCoreProc`，不同 DP ranks 各自调度请求，但 expert collective
要求它们同步推进。空闲 rank 会执行 dummy batch，见
`vllm/v1/engine/core.py:1745-1811,1925-1961`。

因此 MoE DP 不是“互不相关的完整副本”：

- attention、norm、dense/shared 模块可按 DP 复制；
- routed experts 跨 DP/PCP/TP 宽域放置或切分；
- activation dispatch、dummy forward 和 communicator 会影响阶段峰值。

## 9. MoE TP、EP、EPLB

MoE 的有效展平域保持为：

```text
flatten_size = DP * PCP * TP
flatten_rank = dp_rank * PCP * TP + pcp_rank * TP + tp_rank
```

源码：`vllm/model_executor/layers/fused_moe/config.py:1098-1107`。

当 `enable_expert_parallel=false`：

- `ep_size=1`；
- expert tensors 在展平域上做 tensor sharding；
- 每 rank resident bytes 取决于 intermediate dimension、量化 padding 和 loader。

当 `enable_expert_parallel=true`：

- `ep_size=flatten_size`；
- MoE 内部 TP 设为 1；
- 每 rank 持有完整的本地 routed expert 子集。

选择逻辑见 `vllm/model_executor/layers/fused_moe/config.py:1190-1237`。

EPLB 要求 EP 已启用，并可加入 redundant experts。其独立 EPLB group 用于避免与
forward collective 冲突，见 `vllm/distributed/parallel_state.py:1917-1938`；
communicator 可为 torch Gloo/NCCL、NIXL 或 PyNCCL，并可能有 CPU/GPU staging，见
`vllm/config/parallel.py:56-110`。规划器必须把冗余 expert resident bytes 和迁移
峰值作为独立预算。

### 9.1 v0.25.0 sequence-parallel MoE 变更

`FusedMoEParallelConfig.use_all2all_kernels` 从仅 `DP > 1 and EP` 扩展为
`EP and (DP > 1 or sequence_parallel)`，见
`vllm/model_executor/layers/fused_moe/config.py:1035-1042`。

但 release 末又临时恢复 `ParallelConfig.use_sequence_parallel_moe` 对 `DP > 1` 的
限制，见 `vllm/config/parallel.py:633-656`。这是高风险的“底层能力与顶层自动启用
条件不完全一致”状态；planner 不应自行推导 TP-only SP 可用，而应以 resolved
config 和模型路径为准。

## 10. DCP

DCP 不增加 worker，而是把每个 TP group 切成连续 DCP subgroups：

```text
TP % DCP == 0
number of DCP groups per TP group = TP / DCP
```

源码：`vllm/config/parallel.py:498-512`、
`vllm/distributed/parallel_state.py:1813-1833`。

Full attention cache 的本地最大 context 按 `DCP * PCP` 缩小，见
`vllm/v1/kv_cache_interface.py:237-245`。DCP 的目标是当 TP 已超过 KV head 数时，
进一步沿 token 维切 cache，而不是继续复制同一个 KV head。

GQA/MQA 约束包括：

- `TP > total KV heads`；
- `DCP <= TP / total KV heads`；
- `Q heads per KV head` 能被 DCP 整除。

源码：`vllm/config/model.py:1231-1254`。attention implementation 还必须能返回
decode softmax LSE，见 `vllm/v1/worker/cp_utils.py:14-44`。Sliding-window cache
仍显式拒绝 DCP，见 `vllm/v1/kv_cache_interface.py:568-577`；多 block-size hybrid
groups 也不支持 context parallel，见
`vllm/v1/core/kv_cache_utils.py:607-638`。

结论：DCP 是条件支持，不是只要公式整除就能使用。

## 11. PCP

PCP 与 DCP 的核心差异：

- PCP 乘入 worker world size；DCP 不增加 worker；
- PCP 有独立 process group；
- PCP 参与 MoE 展平域；
- cache spec 的 local context 会按 `PCP * DCP` 缩小。

但 `v0.25.0` 全仓只有 attention 基类的：

```python
supports_pcp: bool = False
```

以及运行时对它的 assert，没有任何内置 implementation 设为 true。证据：

- `vllm/v1/attention/backend.py:782-805`；
- `vllm/v1/worker/cp_utils.py:14-44`。

官方 `docs/serving/context_parallel_deployment.md:8-17` 仍将 partial-query/full-KV
和 partial-query/partial-KV 两条路线标为 active development。

因此 MVP 应继续将 `PCP > 1` 标记为 `unsupported/experimental`，阻止“可运行”
结论和可执行导出。配置/group/cache 骨架不能当作运行支持证据。

## 12. KV cache ownership 与容量

普通请求进入一个 EngineCore 后，其 block table 和 KV cache 由该 engine 的
scheduler 管理。普通 DP rank 之间不会自动共享请求 KV；DCP/PCP 才是一个 model
world 内的 context sharding。

单层 full-attention page payload 仍由以下因素决定：

```text
block tokens
* local KV heads or latent width
* (K head size + V head size)
* cache dtype bytes
* alignment / quant scale / page padding
```

不能只使用 `2 * layers * heads * head_dim * tokens * dtype`，因为 MLA、sparse
indexer、hybrid Mamba/state、NVFP4/FP8 scale 和 page padding 均可能改变布局。

`v0.25.0` 的 KV 配置流程会：

1. 合并 worker specs；
2. 按 PP stage 投影 cache groups；
3. 对每个 worker 检查 max-model-len；
4. 各自按 available bytes 生成 blocks；
5. 全 ranks 取最小 block 数。

证据：`vllm/v1/core/kv_cache_utils.py:2005-2153`。

## 13. activation、Graph、runtime 和通信显存

### 13.1 vLLM 自己也不使用纯理论公式

GPU worker 会以最大 profile workload 做 dummy forward，测量：

- model weights；
- PyTorch peak increase，作为 peak activation；
- non-Torch increase；
- 可选 CUDA Graph memory estimate。

然后从 `gpu_memory_utilization` 请求预算中扣除已测得的权重、activation 和
non-Torch 增量，得到 KV bytes。Graph 估值仅在
`VLLM_MEMORY_PROFILER_ESTIMATE_CUDAGRAPHS` 开启时参与这次预扣，见
`vllm/v1/worker/gpu_worker.py:430-541`。

这证明 activation、allocator、kernel workspace 和 Graph 不适合只用静态闭式
公式；planner 应使用 exact-version 校准区间或保守预算。

### 13.2 V1 与 V2 Model Runner 的关键差异

V1 runner：

- `profile_cudagraph_memory()` 会建立临时最小 KV、按 capture descriptors 实际
  profile decoder/encoder graph pool，并返回估算值；
- 真正 capture 后也记录 free-memory 差。

源码：`vllm/v1/worker/gpu_model_runner.py:6498-6644,6647-6712`。

V2 runner：

- `profile_cudagraph_memory()` 固定返回 0；
- 真正 `capture_model()` 才以 capture 前后 free memory 得到 graph bytes。

源码：`vllm/v1/worker/gpu/model_runner.py:680-727`。

`v0.25.0` 的默认 V2 选择已扩展到多数非 MoE generate 模型，同时排除 hybrid 和
attention-free，见 `vllm/config/vllm.py:518-574`。因此同一模型/命令在 0.24 与
0.25 之间可能换 runner，进而改变 persistent buffers、Graph 预算时机和经验峰值。

### 13.3 还需计入的并存项

- TP/PP/EP/DCP collective workspace；
- all2all dispatch/combine buffer；
- DBO 的双 microbatch activation；
- PP intermediate tensors；
- sampler/logits temporary；
- speculative draft/verification buffers；
- connector registration、send/recv/staging；
- allocator fragmentation 和安全余量。

这些项必须按 phase 和并存关系计账，不能把所有最大值无条件相加。

## 14. 多机和 rank 放置

CLI 公开 `--nnodes`、`--node-rank`、master address/port 和 device IDs，见
`vllm/engine/arg_utils.py:958-1008`。多机参数验证和 DP rank 推导见
`vllm/engine/arg_utils.py:1934-1977`。

默认连续 ranks；TP 是最快变化维，源码建议相邻 ranks 位于同一高速互联机器，见
`vllm/distributed/parallel_state.py:1730-1741,1788-1801`。通常应优先把 TP 放节点
内，再让 PP 跨节点，但 planner 应按实际每机卡数和 rank mapping 解析，不把建议
硬编码成永远成立。

Multiprocessing executor 明确要求
`world_size = TP * PP * PCP`，见
`vllm/v1/executor/multiproc_executor.py:103-123`；output rank 是最后 PP stage 的第
一个 TP/PCP worker，见同文件 `495-509`。

## 15. PD 分离

vLLM 的 disaggregated prefill 是两个独立 vLLM instances：一个 prefill，一个
decode，通过 connector 转移 KV 和结果。两个 instance 可各自采用不同 TP/PP；
每个 process 有对应 worker connector，scheduler process 也有 scheduler connector。
见 `docs/features/disagg_prefill.md:1-16,52-88`。

对显存规划器的含义：

- Prefill、Decode 必须分别建立 worker/rank/cache/Graph 账本；
- 两边容量不能相加成单个 cache pool；
- connector 会改变 CUDA Graph 兼容性，部分 connector 会强制 PIECEWISE，见
  `vllm/config/vllm.py:1293-1318`；
- connector 的 GPU send/recv/staging、注册内存和异步并存峰值不能从 core KV
  payload 公式推出；
- PD 在官方文档仍标为 experimental，connector 必须做 exact-version capability
  和校准。

## 16. 相对 v0.24.0 的关键变化

| 主题 | v0.25.0 结论 | 规划器影响 |
|---|---|---|
| TP/PP/DP/PCP/DCP group | 核心布局未变 | 旧拓扑结论可作线索，但需绑定新 tag |
| PCP | 仍无内置 implementation 支持 | 继续阻止 PCP > 1 |
| QKV sharding | head/KV replication 语义未变；loader/fuser API 调整 | 公式不变，但 checkpoint loader 适配需复验 |
| MoE domain | 仍为 `DP*PCP*TP` | expert resident 规则保持 |
| MoE sequence parallel | all2all 底层放宽，顶层 release 又临时保留 DP>1 gate | 标记高风险，不能自行开启 TP-only SP |
| Model Runner V2 | 默认覆盖多数 dense generate 模型 | runtime/Graph/activation 校准必须区分 runner |

## 17. 高风险断言清单

以下结论在实现 planner adapter 前必须由测试或第二次源码审阅锁定。

1. **不要无条件使用 `world_size_across_dp`。** external launcher 下可能二次乘
   DP；应以 resolved launcher world 为准。
2. **不要把 CLI 字段当支持证据。** PCP 就是明确反例。
3. **不要把 `total weights / TP / PP / EP` 当每卡权重。** 必须按组件 shape、
   PP stage、KV replication 和 expert placement 计算。
4. **不要把 Dense DP 与 MoE DP 共用 ownership。** 前者独立，后者 expert
   collective 跨 DP。
5. **不要假设所有 v0.25.0 模型使用同一个 runner。** runner resolution 会改变
   Graph 和 runtime 预算。
6. **不要把 V2 的 Graph profile=0 解释成 Graph 不占显存。** 这是 profile API
   尚未估算，真正 capture 仍测到 bytes。
7. **不要把 DCP 只当除法。** backend LSE、GQA head、sliding-window、hybrid
   cache、MTP 和 connector 都有兼容边界。
8. **不要把 PP layer 数平均后结束。** embedding/head、量化 padding、各 stage
   cache groups 和 min-block 收敛会形成最坏 rank。
9. **不要把 EP 当额外 GPU 乘数。** 在 vLLM 中它使用展平的现有 DP/PCP/TP
   ranks，并改变 expert 权重放置。
10. **不要把 PD connector staging 归入纯 KV payload。** 它是版本/backend 相关
    经验项，且可能与 capture/transfer 并存。

## 18. 给显存规划器的解析输出建议

本审计不确定最终 schema，但 vLLM adapter 至少应解析出：

```text
resolved framework version and model runner
resolved launcher / executor / global world size
physical worker -> DP, PP, PCP, TP, DCP, EP coordinates
PP stage layer ownership
component weight placement and exact local shapes
cache spec per local layer and cache partition ownership
scheduler/cache partition ownership
Graph mode, capture sizes and runner-specific budget source
communication/all2all/connector capability and empirical budget key
unsupported or unverified combinations with source reason
```

共享显存账本可以接收“某物理 GPU 上某组件占多少 bytes、在哪个 phase 并存”；
rank/group 形成、QKV 复制、Dense/MoE DP 分流、MoE 展平域、DCP/PCP capability、
runner 和 connector 语义必须保留在 vLLM `v0.25.0` adapter 中。

## 20. 审计结论

vLLM `v0.25.0` 仍适合采用“framework/version adapter 解析真实 worker、组件放置和
cache ownership，再交给共享物理 GPU 显存账本”的边界。

相较 `v0.24.0`，并行坐标的骨架没有发生推翻式变化，但 Model Runner V2 的默认
范围、Graph 预算行为、MoE sequence-parallel 边界、KV/connector 细节已经足以让
旧 adapter 或旧校准失效。实现必须绑定 `v0.25.0` exact tag，并让 runner、模型、
backend、quantization、connector 和 PD role 共同参与 capability key。
